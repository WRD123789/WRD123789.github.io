多进程服务器端

并发服务器端的实现方法

由于网络程序中数据通信时间比 CPU 运算时间占比更大, 因此, 向多个客户端提供服务是一种有效利用 CPU 的方式. 能同时向多个客户端提供服务的并发服务器的实现模型和方法如下:

  • 多进程服务器: 通过创建多个进程提供服务.
  • 多路复用服务器: 通过捆绑并统一管理 I/O 对象提供服务.
  • 多线程服务器: 通过生成与客户端等量的线程提供服务.

本章主要讨论多进程服务器.

进程

进程的定义如下: 占用内存空间的正在运行的程序. 假如你下载了一个游戏到电脑上, 此时的游戏不是进程, 而是程序. 只有当游戏被加载到主内存并进入运行状态, 这时才可称为进程. 如果同时运行多个游戏, 则会生成相应数量的进程, 也会占用相应进程数的内存空间.

从操作系统的角度来看, 进程是程序流的基本单位, 若创建多个进程, 则操作系统将同时运行它们 (并发或并行).

进程 ID

无论进程是如何创建的, 所有进程都能从操作系统处分配到 ID. 此 ID 称为 “进程 ID”, 其值为大于 2 的整数, 因为 1 要分配给操作系统启动后的首个进程, 因此用户进程无法得到 ID 值 1.

在 Linux 中可以通过 ps 指令查看当前运行的所有进程.

创建进程的方法很多, 此处只介绍用于创建多进程服务器端的 fork 函数:

一个进程可以使用 fork 创建一个新的进程. fork 创建的新进程的内存内容 (包含进程的指令、数据和堆栈) 和文件描述符的表单与调用 fork 的进程 (称为父进程)的内存内容完全相同, 创建的新进程称为子进程. 如果 fork 调用成功, 则 fork 系统调用在父进程中返回子进程的 PID (必然大于 0), 在子进程中返回 0; 调用失败则返回 -1.

僵尸进程

进程的工作完成后 (执行完 main 函数中的程序后) 应被销毁, 但有时这些进程将变成僵尸进程, 占用系统中重要资源. 这种状态下的进程称作「僵尸进程」, 这也是给系统带来负担的原因之一.

子进程的终止方式如下:

  • 传递参数并调用 exit 函数.
  • main 函数中执行 return 语句并返回值.

向 exit 函数传递的参数值和 main 函数的 return 语句返回的值都会传递给操作系统. 而操作系统不会销毁子进程, 直到把这些值传递给产生该子进程的父进程, 由父进程来回收子进程的资源. 所以已经终止执行但其父进程尚未对其进行善后处理的子进程称为僵尸进程.

操作系统不会主动将上述值传递给父进程, 只有父进程主动发起请求 (函数调用) 时, 操作系统才会传递该值. 换言之, 如果父进程未主动要求获得子进程的结束状态值, 操作系统将一直保存, 并让子进程长时间处于僵尸进程状态.

并且即使父进程终止执行, 也不会直接销毁子进程, 而是让 init 进程接管该子进程, 此时该子进程称为孤儿进程. 如果此时子进程终止执行, 会由 init 进程来回收相关的资源.

信号处理

父进程往往与子进程一样繁忙, 因此不能一直调用 waitpid 函数等待子进程终止.

子进程终止的识别主体是操作系统, 因此, 若操作系统能把子进程终止的信息告诉正忙于工作的父进程, 将有助于构建高效的程序. 此时父进程将暂时放下工作, 处理子进程终止相关事宜.

处理子进程需要引入信号处理机制. 此处的 “信号” 是指在特定事件发生时由操作系统向进程发送的消息. 另外, 为了响应该消息, 执行与消息相关的自定义操作的过程称为 “信号处理”.

进程向操作系统 “注册信号” 后, 当进程收到子进程终止的信号时, 会请求操作系统调用特定函数.

从上图可以看出, 每当有客户端请求服务 (请求连接) 时, 回声服务器端都创建子进程以提供服务. 为了完成这些任务, 需要经过如下过程:

  1. 回声服务器端 (父进程) 通过调用 accept 函数受理连接请求.
  2. 将得到的文件描述符传递给子进程 (调用 fork 函数即可完成, 因为 fork 函数会复制父进程的文件描述符表单).
  3. 子进程利用父进程传递过来的文件描述符提供服务.
