Git 介绍

Git 是一个分布式版本控制系统, 而不是集中式的版本控制系统, 即每个开发人员的电脑都存储整个项目的历史 (所有提交记录), 我们将整个项目的历史称为 “存储库”.

Git 的主要组成部分如图所示:

版本控制系统

版本控制系统是跟踪文件随时间变化的工具, 它允许你查看或是恢复到以前的文件状态.

例如: 文档的撤销命令和查看修改历史的功能.

在编码环境中，版本控制系统可以跟踪代码修订的历史, 从代码的当前状态一直追溯到第一次跟踪. 这允许用户引用他们工作的旧版本, 并与其他人(如开发人员同事)共享代码更改.

对象

Git 中的对象如下所示:

  • Blob 对象: 在 git 中存储文件内容的对象. Blob 相比普通文件不包含元数据 (例如: 文件创建时间等), Blob 只有内容 (数据的二进制流). git 中的 Blob 通过 SHA-1 哈希值唯一标识 (对于相同内容的 Blob, 即使在两台不同的电脑上, 产生的 SHA-1 哈希值仍然相同).

  • Tree 对象: 一个目录列表, 引用其他 Blob 和 Tree 对象, 并存储其他对象对应的名字. 用 SHA-1 哈希值唯一标识, 通过其他对象的 SHA-1 引用它们.

  • Commit 对象: git 中的一个快照, 包含指向主要 Tree 对象的指针、元数据 (提交者、提交信息、提交时间) 和该提交的所有父节点 (同样通过它们的 SHA-1 引用它们). Commit 对象同样用 SHA-1 哈希值唯一标识.

跟踪与未跟踪文件

Git 不会跟踪文件夹中的所有文件, 为了保存文件的修改历史, 你需要去跟踪它.

已跟踪文件是指那些 git 已经知道的文件, 要么已经在上次的提交中, 要么已经被放入暂存区.

未跟踪文件是指工作目录中除已跟踪文件之外的文件, 即它们即不在上一次提交中, 也不在暂存区中.

提交记录

Git 仓库中的提交记录保存的是你工作目录下的所有跟踪文件的完整副本.

为了保证提交记录尽可能轻量, 你每次提交时, 它并不会盲目地复制整个目录. 条件允许的情况下, 它会将当前版本与仓库的上一个版本进行对比, 将所有的差异打包在一起作为一个提交记录, 而对于内容相同的部分, Git 不会再次存储文件, 而是保存一个指向存储相同内容的文件的链接, 即相同内容的多个文件只存储一次.

分支

分支通过 Commit 对象的 SHA-1 哈希值引用提交.

创建分支允许你跟踪一个代码的多个不同的版本, 一旦你完成了一个部分的工作可以将它加入到其余代码中, 分支使你能够在代码的不同版本之间切换和合并.

图中有两个分支, 一个是 master 分支, 一个是 feature 分支.

默认情况下, 初始分支一般称为 master, 其他分支可以随意命名, 一般用描述性的名字, 方便记住它包含的提交内容.

使用分支的原因如下:

  • 你可能希望对现有代码进行重构, 但这会破坏项目的其他部分, 同时你希望能够处理代码的其他部分或你有合作伙伴, 你不希望影响他们.
  • 你希望开始处理项目的一个新部分, 但是你不清楚你的更改是否会起作用.
  • 你正在和你的合作伙伴一起工作, 但你不想把你的代码和你的伙伴的工作搞混.

HEAD 指针

HEAD 指针引用你当前的分支作为你的工作目录, HEAD 指针既可以指向提交, 也可以指向分支名 (指向某一分支中提交的指针), 可以移动 HEAD 指针, 以便查看文件的不同版本, 简单来说就是 HEAD 指针指向的是当前分支中的哪个提交 (版本), 当前工作目录中的文件就是哪个版本.
