中断

中断硬件

中断发生时, 操作系统需要保存当前的工作, 处理中断, 处理完成之后再恢复之前的工作, 这和系统调用、Page Fault 的机制相同.

RISC-V 中有很多与中断相关的寄存器:

  • SIE 寄存器: 这个寄存器中有一位 (E) 专门针对外部设备的中断; 有一位 (S) 专门针对软件中断; 有一位 (T) 专门针对定时器中断.
  • SSTATUS 寄存器: 这个寄存器有一位 (SIE 位) 来打开或关闭中断.
    • 每个 CPU 核都有独立的 SIE 和 SSTATUS 寄存器, 除了可以通过 SIE 寄存器来单独控制特定的中断, 还可以用 SSTATUS 寄存器的 SIE 位来控制所有的中断.
  • SIP 寄存器: 用来表示当前存在哪些可以被处理的中断, 每个中断对应一位, 如果某一位为 1, 表示对应的中断待处理.
  • …

但是中断和系统调用又有些地方不一样, 中断和系统调用主要的区别如下:

  • 异步. 当硬件生成中断时, Interrupt Handler 与当前运行的进程在 CPU 上没有任何关联, 而系统调用通常发生在运行进程的上下文中.
  • 并行. 对于中断来说, CPU 和生成中断的设备是并行运行.
    • 例如网卡自己独立的处理来自网络的 packet, 然后在某个时间点产生中断, 但是同时, CPU 也在运行.
  • 程序设备: 主要关注外部设备, 例如网卡、UART, 而这些设备需要被编程. 每个设备都有一个编程手册, 就像 RISC-V 有一个包含了指令和寄存器的手册一样. 设备的编程手册包含了它有什么样的寄存器, 它能执行什么样的操作, 在读写控制寄存器的时候, 设备会如何响应.

外部设备的中断主要来自主板上的设备. 下图是一个 SiFive 主板:

  • 大量的设备 (以太网卡、MicroUSB、MicroSD等) 连接在或者可以连接到这个主板上.
  • 主板上的各个线路将外设和 CPU 连接在一起.

下面是有关 SiFive 处理器的文档, 图中的右侧是各种各样的设备. 类似于读写内存, 通过向相应的设备地址执行 load/store 指令, 就可以完成对相应设备的编程.

  • 所有的设备都连接到处理器上, 处理器是通过 Platform Level Interrupt Control (PLIC) 处理设备中断. PLIC 会管理来自于外设的中断. 如果我们再进一步深入的查看 PLIC 的结构图:

    • 从左上角可以看出, 我们有 53 个不同的来自于设备的中断. 这些中断到达 PLIC 之后, PLIC 会路由这些中断. 图的右边是 CPU 的核, PLIC 会将中断路由到某一个 CPU 的核. 如果所有的 CPU 核都正在处理中断, PLIC 会保留中断直到有一个 CPU 核可以用来处理中断. 所以 PLIC 需要保存一些内部数据来跟踪中断的状态.
    • 在 Xv6 中 (在其他操作系统中不一定, 这取决于你如何对 PLIC 进行编程), 当某一外设触发中断时, 所有的 CPU 核都能收到中断, 但只有一个 CPU 核会要求接收中断.
    • 具体流程是:
      1. PLIC 通知当前有一个待处理的中断.
      2. 其中一个 CPU 核会要求接收中断, 这样 PLIC 就不会把中断发给其他的 CPU 核处理.
      3. CPU 核处理完中断之后, CPU 会通知 PLIC.
      4. PLIC 将不再保存中断的信息.

Xv6 中断设置

在 start (kernel/start.c) 中:

  • 在该函数中完成以下工作:
    • 将所有的中断都设置在内核模式.
    • 设置 SIE 寄存器来接收外设中断、软件中断和定时器中断.
    • 初始化定时器.

在 main (kernel/main.c) 中:

  • 首先会进行控制台的初始化, 调用 consoleinit(kernel/console.c) 来完成, consoleinit 调用 uartinit (kernel/uart.c) 来初始化 UART:
    • 该函数的主要工作是:
      • 写入相关的控制寄存器.
      • 配置好传输的波特率 (串口线的传输速率).
      • 重置 FIFO 缓冲区.
      • 开启接收中断和发送完成中断.
    • 当 UART 接收到一个字节的输入时, 就会产生接收中断; 当 UART 完成一个字节的发送时, 就会产生发送完成中断.
  • 然后是 PLIC 的初始化:
    • PLIC 未初始化, 中断就无法被 CPU 感知到.
    • PLIC 与外设一样, 也占用了一个 I/O 地址 (0xC000_0000).
    • 两行代码实际上是设置了 PLIC 会接收哪些中断, 进而将中断路由到 CPU.
      • 第一行设置了 PLIC 接收来自 UART 的中断.
      • 第二行设置了 PLIC 接收来自 I/O 磁盘的中断.
  • 然后对每个 CPU 核调用 plicinithart:
    • 每个核调用该函数来表示想要接收哪些外设中断.
    • 由于我们忽略中断的优先级, 所以将优先级设置为 0.
  • 到目前为止, 我们有了生成中断的外部设备, 我们有了 PLIC 可以传递中断到单个 CPU, 但是 CPU 还没设置好接收中断, 因为我们还没设置好 SSTATUS 寄存器, 所以在 main 函数的最后调用 scheduler 函数:
    • 该函数主要是运行进程, 但是在运行进程之前, 会执行 intr_on 函数来使 CPU 能接收中断:
      • 该函数设置 SSTATUS 寄存器, 打开中断标志位.
  • 现在中断已经被完成打开了.

计时器中断

Xv6 使用了计时器中断, 维护计时器的运行, 并且能因此引入基于时间片的进程切换机制. 计时器中断产生时, 首先在机器模式下产生一个软件中断, 提醒内核及时处理计时器到时这一事件; 在后续内核进行user trap 或 kernel trap 的过程中, 如果中断开放, 就能够注意到这一软件中断, 从而执行时间片到期时的相应逻辑事件 (如调度) .

  • 计时器中断在机器模式下处理, 而不是内核模式. 在 RISC-V 的机器模式下执行时, 页表将被禁用, 而且有专门的一套寄存器 (机器模式下的寄存器以 m 开头命名, 内核模式下的以 s 开头) .计时器中断要在机器模式下执行的因为是:
    • 在机器模式下, 中断处理程序可以访问所有的物理内存, 包括操作系统的内核地址空间, 从而可以更好地管理系统资源, 并进行更加高效的进程调度和管理.
    • 在机器模式下, 中断处理程序也可以直接操作硬件设备, 例如计时器和时钟设备等, 从而更加灵活地实现系统的定时、延时等功能.
    • 在机器模式下, 中断处理程序可以使用一些特殊的机器模式相关寄存器, 例如 sstatus、scause、sepc 等, 这些寄存器能够提供更多的中断信息和调试信息, 方便开发人员进行系统调试和错误处理.

计时器中断来自于硬件的计时器芯片, 每个CPU都有一片, xv6负责周期性地对这些计时器芯片重新编程, 以重置并开始新一轮计时.

当 Xv6 刚启动、还处于机器模式时, 就进行了计时器的初始化 (kernel/start.c) :

  • 该计时器初始化函数的主要工作有:
    • 对 CLINT 进行编程, 使其在一定时延后产生中断.
    • 设置类似 Trapframe 的容器 scratch, 用于在机器模式下保存寄存器和 CLINT 的地址.
    • 设置 mtvec, 使得机器模式的 Trap 能够跳转到 timervec, 并且开放计时器中断.

计时器中断可以在任何时刻发生, 不管是在执行用户或内核代码. 即使内核正在临界区中执行, 也不应该关闭计时器中断.

时间片到期时, Xv6 会陷入机器模式, 跳转到我们预先在寄存器 mtvec 中设置好的计时器中断处理程序 timervec 中:

  • 该汇编函数的主要工作如下:
    • 保存一些在 Xv6 启动阶段就设置好的机器模式相关的寄存器到 scratch 中.
    • 设置 CLINT 以便产生下一次计时器中断.
    • 让 CPU 发出软件中断.
    • 恢复寄存器并返回.

通过发出软件中断, CPU 将处理计时器中断的任务交付给了内核, 现在内核可以按照与普通中断相同的 Trap 机制来处理该中断, 显然内核也可以屏蔽该软件中断 (例如当执行类似 acquire 的原子操作时, 要关闭中断) . 处理该软件中断的流程在 devintr 中.

中断相关的并发

中断相关的并发包含以下几个方面:

  • 设备与 CPU 是并行运行的, 这里的并行称为生产者-消费者模式.
    • 例如: 当 UART 向控制台发送字符时, CPU 会返回继续执行 Shell, 此时设备和 CPU 是并行运行的.
  • 驱动的 bottom half 和 top half 部分是并行运行的.
    • 例如: Shell 会在传输完提示符 “$” 之后再调用 write 系统调用传输空格字符, 代码会走到 UART 驱动的 top half 部分 (注: uartputc函数), 将空格写入到寄存器中; 但是同时在另一个 CPU 核中, 可能正好收到来自于 UART 的中断 (之前发送完 “$” 会触发发送完成中断), 进而执行 UART 驱动的 bottom half 部分, 查看相同的缓冲区.

我们主要关注生产者-消费者模式, 这是驱动中的非常常见的现象:

  • 首先在驱动中会有一个缓冲区, 在该例中, 缓冲区的大小为 32 字节, 并且有两个指针, 分别是读指针和写指针

    • 如果两个指针相等, 那么缓冲区是空的.

  • 此时用户进程 (例如 Shell) 调用 uartputc 函数, 则会将字符, 例如提示符 ‘$’ 写入到写指针的位置, 并使写指针往后移动一个单位

    • 这就是生产者对缓冲区的操作.

  • 生产者可以一直写入数据, 直到写指针 + 1 等于读指针时, 此时缓冲区已经满了. 当缓冲区满的时候, 生产者必须停止运行.

    • 在 uartputc 函数中, 当缓冲区满的时候, 会调用 sleep 让出 CPU.

  • 中断处理程序, 也就是 uartintr 函数, 在这种情况下是消费者, 每当有一个中断, 并且读指针落后于写指针, uartintr 函数就会从读指针中读取一个字符再通过 UART 设备发送, 并且将读指针加 1.

    • 当读指针追上写指针, 也就是两个指针相等时, 缓冲区为空, 此时不用做任何操作.

中断的演进

当 Unix 刚被开发出来的时候, Interrupt 处理还是很快的. 这使得硬件可以很简单, 当外设有数据需要处理时, 硬件可以中断 CPU 的执行, 并让 CPU 处理硬件的数据.

现在, 中断相对处理器来说变慢了. 由于从中断发生到真正的处理中断处理中断数据之间需要很多步骤, 当一个设备在高速的产生中断时, 处理器很难跟上.

  • 例如: 你有一个千兆网卡, 这个网卡收到了大量的小包, 网卡每秒可以生成 1.5Mpps, 这意味着每一个微秒, CPU 都需要处理一个中断, 这就超过了 CPU 的处理能力.

这里的解决方法时使用 Polling: 除了依赖于中断, CPU 可以一直读取外设的控制寄存器来检查是否有数据.

  • 例如: 对于 UART 来说, 我们可以一直读取 RHR 寄存器来检查是否有数据.

CPU 不停地轮询设备, 直到设备拥有数据, 这减少了处理中断的开销, 但是 CPU 在不停地轮询外设的控制寄存器时并不能运行任何程序.

所以对于一个慢设备, 不应当采用 Polling 而应当采用中断; 对于一个快设备, 应当采用 Polling 而不应当采用中断.

对于一些精心设计的驱动, 它们会在 Polling 和中断之间动态切换.
