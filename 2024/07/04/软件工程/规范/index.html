规范

规范

规范是团队合作的关键. 没有规范, 委托实施方法的责任是不可能的. 规范起着合同的作用: 实施者负责满足合同, 使用该方法的客户端可以依赖合同. 实际上, 与真正的法律合同一样, 规范对双方都有要求: 当规范具有前提条件时, 客户也有责任.

规范对于模块的客户端是有益的, 有了规范, 你无需阅读模块的代码就可以理解模块的作用, 这使得模块更容易理解.

规范对于方法的实现者也是有益的, 因为它们给了实现者在不告知客户端的情况下更改实现的自由. 规范也可以使代码更快, 因为规范可以排除方法被调用的某些状态, 对输入的这种限制可能允许实现者跳过昂贵的检查, 而使用更有效的实现.

规范作为合同起到客户和实施者之间的防火墙作用. 它保护客户免受模块工作细节的干扰: 作为客户, 如果你有模块的规范, 就不需要阅读模块的源代码. 它也保护实施者免受模块使用细节的干扰: 作为实施者, 你不必询问每个客户他们计划如何使用模块. 这种防火墙导致解耦, 允许模块代码和客户代码独立地进行更改, 只要这些更改遵守规范 - 每个人都遵守自己的义务.

规范结构

方法的规范包括几个部分:

  • 方法签名: 包含方法名称、参数类型、返回类型和引发的异常.
  • 需求 (requires): 描述对参数的附加限制.
  • 结果 (effects): 描述方法的返回值、异常和其他结果.

这些部分共同构成了该方法的前置条件和后置条件

  • 前置条件是客户端 (方法的调用者) 的义务. 它是对调用方法时状态的一种条件, 该条件表示该方法被调用时系统或程序必须处于某种特定的状态下, 以确保方法能够正确执行. 前置条件包括方法签名中参数的数量和类型. 附加条件需要写在 requires 中. 包括:
    • 缩小参数范围. (例如: x >= 0 表示 int 类型的参数 x 必须是非负的 int)
    • 参数之间的交互. (例如: val 在 arr 中只出现一次)
  • 后置条件是方法实施者的义务. 它包括 Java 可以静态检查的部分: 返回类型和声明的检查异常. 附加条件可以写在 effects 中, 包括:
    • 返回值与输入的关系如何.
    • 会引发哪些异常, 以及何时引发异常.
    • 对象是否以及如何变化.

通常情况下, 后置条件是指在调用方法之后, 程序状态需要满足的条件, 前置条件是假设在方法调用之前已经成立的条件. 这种关系可以表达为逻辑蕴含式: 如果前置条件在方法调用时成立, 则当方法完成后必须满足后置条件. 如果前置条件在方法调用时不成立, 那么实现就不受后置条件的约束. 它可以任意操作, 包括不返回任何结果、抛出异常、返回任意结果、进行任意修改等.

Java 规范

有些语言将前置条件和后置条件作为语言的基本组成部分, 作为运行时系统 (甚至编译器) 可以自动检查的表达式, 以强制执行客户端和实现者之间的合同.

Java 并没有完全采用这种方法, 但它的静态类型声明实际上是方法的前置条件和后置条件的一部分, 这部分条件由编译器自动检查和强制执行. 合同的其余部分 - 那些不能作为类型来编写的部分, 必须在方法前的注释中进行描述, 通常需要人工来检查和保证它的正确性.

Java 有一个文档注释约定, 称为 Javadoc, 其中参数由 @param 子句描述, 结果由 @return 子句描述. 你应该尽可能将前置条件放入 @param , 将后置条件放入 @return.

不允许空引用

基本数据类型变量不允许被赋值 null, 而非基本数据类型变量允许被赋值变量, 但是不能调用指向 null 的变量的方法.

空引用容易导致 Bug, 所以大多数方法会避免它们. 作为一般约定, 参数和返回值不允许使用 null 值, 除非规范明确另有说明. 因此每个方法在其参数中对对象和数组类型都有一个前置条件, 即它们是非空的 - 包括数组、集合等的元素. 每个可以隐式返回对象或数组类型的方法都有一个后置条件, 即它们的值是非空的, 同样包括集合类型的元素.

如果一个方法在参数中允许空值, 它需要显式地声明它, 或者如果它可能在结果中返回空值, 它应该显式地声明它.

如果避免使用 null, 有时仍然需要一个参数或返回值来表示值的缺失. 解决这个问题的一个好工具是 Optional<T>. 它相当于一个长度为 1 的 List<T>. 它要么只包含一个 T 类型的元素, 要么为空. isPresent() 方法可以用来测试它是否为空, get() 和 getOrElse() 方法可以获取它包含的值. 在规范允许存在缺失值的情况下, Optional<T> 能够清晰地表达规范的意图.

这样的测试用例才能独立于实现且符合规范, 即使你更改了方法的实现, 也无需重新测试用例.

对于我们编写的一个方法的单元测试, 它不应该因为其他方法不满足规范而失败.

好的集成测试应该确保不同模块的规范兼容, 也就是说, 当一个方法作为调用者调用另一个方法时, 两个方法的输入和输出值应当符合对方期望的. 这有助于确保整个程序的功能正确性.

当然, 集成测试不能取代单元测试, 因为单元测试的目标是对程序中的单个模块进行测试, 可以更加精细地控制测试的输入和输出, 从而发现并排除可能存在的潜在问题. 如果我们只依赖于集成测试, 那么我们可能会忽略程序中单个模块的一些特定场景, 从而导致漏洞或错误隐藏在程序中.

方法的签名 – 它的名称、参数类型、返回类型 是其规范的核心部分, 签名还可能包括方法可能触发的异常.

Java 中, 异常可以分为两种类型:

  • Checked Exceptions: Checked Exceptions 是指那些在编译时需要明确处理的异常. 它们在方法的声明中被列出来, 表示该方法可能抛出这些异常. 当调用这种方法时, 必须要么使用 try-catch 块来捕获这些异常, 要么继续将这些异常抛出. 因为编译器会强制检查这些异常的处理, 所以 Checked Exceptions 常用于处理预期的问题, 例如文件未找到、网络连接失败等.
  • Unchecked Exceptions: 是指那些在编译时不需要强制处理的异常. 这些异常通常是由编程错误导致的, 例如空指针引用、数组越界、类型转换错误等等. 如果发生了这些异常, 程序将抛出一个未处理的异常并终止运行. 因为这些异常是由编程错误引起的, 所以它们不需要在方法的声明中被列出来, 也不需要在每个方法调用处进行 try-catch 处理. 但是, 为了确保程序的健壮性, 我们仍然需要在适当的时候捕获这些异常.

在编写程序时, 我们需要根据实际情况来选择使用 Checked Exceptions 还是 unchecked Exceptions. 如果一个异常是预期可能发生的情况, 我们应该使用 Checked Exceptions 来强制处理这种异常. 如果一个异常是由编程错误导致的, 我们应该使用 unchecked Exceptions 来标识这些问题.

在上述异常层次结构中:

  • Exception 是 Checked Exceptions 的基类. 编译器将静态检查应用于使用这些异常的方法. 必须在可能引发异常时捕捉或声明 Checked Exceptions.
  • RuntimeException 及其子类是 Unchecked Exception. 不必在抛出它们的方法的 throw 子句中声明, 也不必由此类方法的调用者捕捉或声明.
  • Error 及其子类也是 Unchecked Exception. 它们应该被认为是不可恢复的, 所以不应该被代码捕捉.

当你定义自己的异常时, 要么继承 Exception, 要么继承 RuntimeException, 而不应该继承 Error.

当使用 try-catch 语句捕获异常时, 应该尽可能捕获最具体的异常类. 如果希望出现 FileNotFoundException, 那么 catch 语句应该使用该异常. 捕获一个广泛的异常类, 比如 Exception、RuntimeException 或 Error, 对于 Bug 或准备进行更改来说并不安全, 因为它捕获了这些异常的每一个可能的子类, 这可能会干扰静态检查并隐藏 Bug.
