测试

测试

  • 模块是软件系统的一部分, 它可以独立于系统的其他部分进行设计、实现、测试和推理.
  • 规范描述模块的行为. 对于函数, 规范给出了参数的类型以及对它们的任何附加约束. 它还提供了返回值的类型以及返回值与输入的关系.
    • 在 Java 中, 规范由方法签名和上面的注释组成, 注释描述了方法的功能.
  • 模块具有提供其行为的实现和使用该模块的客户端. 对于函数, 实现是方法的主体, 客户端时调用该方法的其他代码. 模块的规范同时约束了客户端和实现.
  • 测试用例是输入的特殊选择, 以及规范所要求的预期输出行为.
  • 测试组是模块的一组测试用例.

测试优先编程

在从头开始设计程序时, 测试优先编程是一个很好的模式. 在测试优先编程中, 你甚至在编写任何代码之前就已经编写了规范和测试. 单一函数的开发按以下顺序进行:

  • 规范: 为函数编写规范.
  • 测试: 编写执行规范的测试.
  • 实现: 编写实现, 即函数主体.

测试优先编程的最大好处是避免 Bug.不要把测试留到开发结束, 当你有一大堆未经验证的代码的时候, 将测试留到最后只会使调试更长、更痛苦, 因为 Bug 可能存在于代码的任何地方.

系统测试

我们希望系统地进行测试, 而不是彻底的、随意的或随机的测试. 系统测试意味着我们以一种合理的方式选择测试用例, 目标是设计一个具有三个理想属性的测试组:

  • 正确: 正确的测试组是规范的合法客户端, 它可以接受规范的所有合法实现. 这使我们可以自由地更改模块在内部的实现方式, 而不必更改测试组.
  • 彻底: 一个彻底的测试组可以发现实现中的实际错误, 这些错误是由程序员可能犯的错误造成的.
  • 小: 一个小型测试组, 编写起来更快, 如果规范发展, 更新起来也更容易. 小型测试组的运行速度也更快, 能够更频繁地运行测试.

分区选择测试

分区选择测试样例旨在将输入空间划分成不同的子域, 并从每个子域中选择代表性的测试样例来测试被测程序, 不相交子域的集合称为分区. 该技术可以帮助测试人员更高效地选择测试用例, 并覆盖输入空间中的不同情况.

分区选择测试样例的具体步骤如下:

  1. 确定输入空间: 首先需要确定被测试程序的输入空间, 即所有可能的输入值的集合.
  2. 划分输入空间: 将输入空间划分成不同的子域, 每个子域包含相似的输入值. 这可以根据输入值的特征 (如数据类型、大小、范围等) 或业务逻辑规则 (如状态转换、输入条件等) 进行划分.
  3. 选择测试用例: 从每个子域中选择测试用例, 以覆盖该子域中的不同输入情况.
  4. 执行测试用例: 使用所选测试用例执行测试, 并记录测试结果.
  5. 分析测试结果: 确定测试用例是否成功捕捉被测程序的缺陷. 如果没有, 可以重新选择测试用例或更改测试策略.

这种方法通过选择不同的测试用例, 并强制测试探索随机测试可能无法到达的输入空间, 从而最大限度地利用有限的测试资源.

黑盒测试与白盒测试

黑盒测试: 只从规范中选择测试样例, 而不是根据函数的实现选择测试用例.

  • 例如:
    • 我们分区和查找 abs、 max 和 multiply 函数中的边界, 而没有查看这些函数的实际代码. 实际上, 按照测试优先的编程方法, 我们甚至还没有为这些函数编写代码.

白盒测试: 在了解函数具体实现的情况下选择测试用例.

  • 例如:
    • 如果实现根据输入选择不同的算法, 那么应该围绕选择不同算法的点进行划分.
    • 如果实现保留了一个内部缓存来记住以前输入的答案, 那么你应该测试重复的输入.
    • 对于 BigInteger.multiply, 当我们最终实现它时, 我们可能已经决定用 int 值表示小整数, 用十进制数列表示大整数. 这个决策将引入新的边界值 (Integer.MAX_VALUE 和 Integer.MIN_VALUE).
  • 当进行白盒测试时, 测试人员可以查看代码实现的细节, 这使得他们有可能根据代码实现的细节编写测试用例. 然而, 有时候编写的测试用例可能只能通过当前的实现方式才能得到正确的结果, 而不是依据规范要求. 这种情况可能会导致测试人员对代码实现的行为做出错误的假设, 从而导致在将来修改代码实现时出现问题. 因此, 在进行白盒测试时, 测试人员需要仔细阅读规范并确保测试用例仅仅检查规范中所要求的行为. 如果规范没有具体说明某些行为, 则测试用例应该是通用的. 不依赖于当前的实现方式. 这样测试人员就可以确保测试用例正确地反映了规范所要求的行为, 而且这些测试用例在将来代码实现的修改下仍然有效.

覆盖率

覆盖率是指在软件测试中, 测试用例覆盖代码中各种情况和路径的程序.

以下是三种常见的覆盖率:

  • 语句覆盖率: 测试用例覆盖代码中每个语句至少一次的情况. 如果一条语句没有被测试用例覆盖到, 那么这个语句就没有得到执行.
  • 分支覆盖率: 测试用例覆盖代码中每个分支的情况. 如果一个条件语句中的每个分支都被测试用例至少一次覆盖到, 那么这个条件语句就可以被认为是完全覆盖了.
  • 路径覆盖率: 测试用例覆盖代码中每个可能的执行路径的情况. 一个程序的执行路径可能有很多条, 路径覆盖率就是指测试用例覆盖了多少个不同的执行路径. 路径覆盖率是三种覆盖率中最严格的一种, 因为它需要测试用例覆盖所有可能的执行路径, 而不仅仅是语句或分支.

一个标准的测试方法是添加测试, 直到测试组达到足够的语句覆盖率. 通常语句覆盖率由代码覆盖率工具来衡量, 该工具计算测试组运行每个语句的次数.

使用代码覆盖率工具, 可以使得白盒测试更容易: 你只需要测量黑盒测试的覆盖率, 并添加更多的测试用例, 直到所有重要的语句都被记录为执行状态.

  • 问: 为什么要测量黑盒测试的覆盖率?
    • 答: 在白盒测试中, 测试人员需要了解代码内部的结构和逻辑, 以编写测试用例并检查代码的内部行为. 但是, 测试人员不能仅仅依靠自己的经验和想象来编写测试用例, 而应该参考软件需求和规格说明书等文档, 确保测试用例覆盖了所有的功能需求和限制条件. 而黑盒测试是基于软件需求和规格说明书, 而不考虑软件内部实现的测试方法. 因此, 在进行白盒测试时, 可以利用已经编写的黑盒测试用例, 以及黑盒测试产生的覆盖率信息, 来辅助编写白盒测试用例. 通过测量黑盒测试的覆盖率, 测试人员可以了解哪些代码路径已经被测试用例覆盖, 哪些代码路径还需要编写额外的白盒测试用例来覆盖. 这种方法可以帮助测试人员快速地定位需要编写的测试用例, 并减少测试人员的工作量. 同时, 还可以提高测试的效率和覆盖率, 从而提高软件的质量.

单元测试与集成测试

单元测试是指对软件中的独立单元进行测试, 例如函数、方法或类等. 单元测试通常是由开发人员编写的, 旨在验证单个单元是否按照预期工作. 在单元测试中, 测试者通常会模拟该单元的依赖项并使用不同的测试用例来验证该单元在各种输入情况下的输出是否正确. 单元测试可以使调试更加容易. 当一个模块的单元测试失败时, 你可以更加确信错误是在该模块中发生的, 而不是在程序的其他地方发生的.

集成测试则是在不同组件之间进行测试, 以验证它们是否可以正常协同工作. 在集成测试中, 测试者会测试软件系统的不同模块之间的交互, 并确保它们可以按照预期协同工作. 集成测试通常由专门的测试人员进行, 测试范围更广, 测试的是整个软件系统而非单个单元. 那么当一个测试失败时, Bug 可能发生在程序的任何地方. 但集成测试依然很重要, 因为程序可能在模块之间的连接处失败.

  • 例如: 一个模块可能期望的输入与实际从另一个模块中获得的输入不同.
    但是, 如果你有一套完整的单元测试使你对单个模块的正确性有信心, 那么就不需要花费过多的时间寻找 Bug.

  • 存根是在测试过程中用来替代某些组件或者模块的一个简单实现, 它可以模拟出组件或模块的某些行为或输出，以便测试其他组件或模块的行为. 在软件测试中, 有些组件或模块可能依赖于其他组件或模块的功能. 如果这些依赖的组件或模块尚未完成或者不可用, 那么就可以使用存根来代替. 存根通常是一个非常简单的程序, 它只实现了被测试组件或模块所需要的一些基本功能, 而其他的功能则直接返回固定的结果. 使用存根可以简化测试过程, 减少测试的时间和成本, 并且提高测试的可靠性.

自动回归测试

回归测试是指在软件系统修改或更新之后, 重新执行之前已经执行过的测试用例, 以确保修改或更新后的软件系统没有破坏原有的功能和性能, 也没有引入新的 Bug. 回归测试的目的是验证软件修改或更新的正确性, 并且保证软件系统在新的版本中依然能够按照原有的功能和性能运行.

自动回归测试是指使用自动化测试工具来执行回归测试的过程. 相比手动回归测试, 自动回归测试能够更加快速地执行测试用例, 并且能够重复执行多次, 以确保软件系统的稳定性和可靠性. 自动化测试工具可以自动执行预设的测试用例, 比较预期结果和实际结果之间的差异, 并生成测试报告. 自动回归测试能够减少测试的人力成本和时间成本, 并且提高测试的可靠性和准确性.

无论何时你发现并修复了一个 Bug, 应当将引发该 Bug 的输入作为测试用例添加到你的测试组中, 这有助于优化你的测试组, 防止重新引入该 Bug.

迭代测试优先编程

迭代测试优先编程是对测试优先编程的完善, 其具体流程如下:

  1. 为函数写一个规范.
  2. 编写执行规范的测试. 当你发现问题时, 对规范和测试进行迭代.
  3. 编写函数的实现. 当你发现问题时, 对规范、测试和实现进行迭代.
    每个步骤都有助于验证前面的步骤.
  • 编写测试有助于理解规范, 规范可能时不正确的、不完整的、模糊的. 尝试编写测试可以在浪费时间实现错误的规范之前及时发现问题.
  • 编写实现有助于发现缺失或不正确的测试, 或者提醒你重新审视和修改规范.

迭代规划:

  • 对于大型规范, 首先只编写规范的一部分, 然后继续测试和实现该部分, 然后使用更完整的规范进行迭代.
  • 对于复杂的测试组, 首先选择一些重要的分区, 然后为它们创建一个小的测试组, 编写一个能通过这些测试的简单实现, 然后在具有更多分区的测试组上迭代.
  • 对于一个复杂的实现, 首先编写一个简单的实现来测试你的规范和测试组, 确保你的规范和测试组是正确的, 然后再编写更难的实现.

在迭代测试优先编程中, 软件开发人员通过快速迭代, 逐步完善软件产品, 以达到更高的质量和更快的交付速度.

迭代测试优先编程要求开发人员拥有不同的思维方式. 相比于解决作业和考试问题时一次性地解决问题, 迭代测试优先编程要求开发人员尽快提供一个初步的解决方案, 并不断地进行改进和优化, 以便有时间进行必要的舍弃和重新工作. 这种方法对于解决困难问题和未知的解决方案空间的问题尤为有效.

迭代测试优先编程方法的一个重要方面是测试优先. 测试应该成为开发过程的一部分, 而不是在开发完成后进行的单独的活动. 测试应该尽早地开始, 以便在整个开发过程中快速发现和修复问题.
