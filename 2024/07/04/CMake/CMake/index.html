CMake

CMake 是一个跨平台的项目构建工具. 常见的项目构造工具还有 Makefile, 但是 Makefile 相比 CMake 更加依赖于当前编译平台, 而且编写工作量更大.

CMake 会根据当前编译平台自动生成本地化的 Makefile 和工程文件, 最后用户只需要调用 make (执行 Makefile 文件中的指令) 即可编译生成可执行文件. 所以可以把 CMake 看成一款自动生成 Makefile 的工具, 其编译流程如下图所示:

  • 蓝色虚线表示使用 Makefile 构建项目的过程.
  • 红色实线表示使用 CMake 构建项目的过程.

根目录: 是整个项目的顶级目录, 它是包含所有源代码、文档、配置文件等的目录.

编译目录/构建目录: 用来存放项目构建过程中生成的文件的地方, 也就是执行 cmake 指令的文件夹.

链接库文件

动态库的链接和静态库的链接是完全不同的:

  • 静态库会在生成可执行程序的链接阶段被打包到可执行程序中, 所以可执行程序启动, 静态库就被加载到内存中.
  • 动态库在生成可执行程序的链接阶段不会被打包到可执行程序中, 当可执行程序被启动并且调用了动态库中的函数的时候, 动态库才会被加载到内存.

在编写程序的过程中, 可能会用到一些系统提供的动态库或者自己制作的动态库或者静态库文件, 在 CMake 中也提供了链接库的指令:

  • link_libraries 指令用于将库链接到稍后添加的所有目标 (一般用于静态库):
    • 库的名字可以是全名 (“libxxx.a”), 也可以是去掉头 (“lib”) 和尾 (“.a”) 后的名字 (“xxx”).
    • 如果库不是系统提供的, 可能存在找不到库的情况, 此时可以指定库的路径: 例如:
  • target_link_libraries 指令指定链接给定目标时要使用的库 (一般用于动态库):
    • target: 指定要链接的库的名字, 该文件可以是一个源文件, 也可以是一个动态库文件, 还可以是一个可执行文件 (必须通过 add_executive 或 add_library 之类的指令创建的).
    • PRIVATE | PUBLIC | INTERFACE: 表示库的访问权限, 默认为 PUBLIC. 如果库名去掉头 (“lib”) 和尾 (“.so”/“.a”), 则默认优先链接动态库.
      • 库的链接具有传递性. 如果库 A 链接了库 B、C, 库 D 链接了库 A, 此时库 D 相当于也链接了库 B、C, 并可以使用库 B、C 中的函数.
      • PUBLIC 声明的库会被链接到 target 中, 并且该库中的函数也会被导出, 提供给第三方 (如果 target 是库文件, 则这里的第三方指的是链接 target 的文件) 使用.
        • 例如: target_link_libraries(A PUBLIC B), target_link_libraries(C PUBLIC A), 此时库 C 中可以调用库 B 中的函数.
      • PRIVATE 声明的库会被链接到 target 中, 并且该库的链接到此为止, 第三方无法知道 target 链接了该库.
        • 例如: target_link_libraries(A PRIVATE B), target_link_libraries(C PUBLIC A), 此时库 C 无法调用库 B 中的函数, 也不知道库 A 链接了库 B.
      • INTERFACE 声明的库不会被链接到 target 中, 只会导出该库中的函数, 提供给 target 使用.
        • 例如: target_link_libraries(A INTERFACE B), 此时库 A 能够使用库 B 中的函数, 但是不知道这些函数来自哪个库.
    • 在 cmake 中指定要链接的库的时候, 应该将指令写到用来生成可执行文件/库的指令的后面 (因为需要指定 target):

显示消息

可以通过 message 显示消息:

  • (无): 重要消息
  • STATUS: 非重要消息
  • WARNING: CMake 警告, 会继续执行
  • AUTHOR_WARNING: CMake 警告 (dev), 会继续执行
  • SEND_ERROR: CMake 错误, 继续执行, 但是会跳过生成的步骤
  • FATAL_ERROR: CMake 错误, 终止所有处理过程
  • CMake 的命令行工具会在 stdout 上显示 STATUS 消息, 在 stderr 上显示其他所有消息. CMake 的 GUI 会在它的 log 区域显示所有消息.

list 操作

list 指令其余功能:

  1. 获取 list 的长度:
    • <list>：当前操作的变量, 值为 list.
    • <output variable>: 变量, 用于存储列表的长度.

CMake 嵌套

如果项目很大, 或者项目中有很多的源码目录，在通过 CMake 管理项目的时候如果只使用一个 CMakeLists.txt, 那么这个文件相对会比较复杂, 有一种化繁为简的方式就是给每个源码目录都添加一个 CMakeLists.txt 文件, 这样每个文件都不会太复杂, 而且更灵活, 更容易维护.

例如:

因为 Linux 的目录是树状结构, 所以嵌套的 CMake 也是一个树状结构, 最顶层的 CMakeLists.txt 是根节点, 其次都是子节点.

CMakeLists.txt 文件中变量的作用域:

  • 根节点 CMakeLists.txt 中的变量全局有效.
  • 父节点 CMakeLists.txt 中的变量可以在子节点中使用.
  • 子节点 CMakeLists.txt 中的变量只能在当前节点中使用.

在一个节点 (CMakeLists.txt 文件) 中使用指令 add_subdirectory 可以指定某个节点为该节点的子节点, 以此来完成父子节点之间关系的建立. 该指令形式如下所示 (后两个参数一般情况下不使用):

  • source_dir: 节点 (CMakeLists.txt 文件) 的路径, 表示这个节点为当前节点的子节点.
  • binary_dir: 指定输出文件的路径, 一般不需要指定, 忽略即可.
  • EXCLUDE_FROM_ALL: 在子路径下的目标默认不会被包含到父路径的 ALL 目标里, 并且也会被排除在 IDE 工程文件之外. 用户必须显式构建在子路径下的目标.

  1. 判断文件或者目录是否存在 如果文件或者目录存在返回 True, 否则返回 False.
  2. 判断是不是目录

  2. 比较两个路径是否相等
    • 相比 if(<variable|string> STREQUAL <variable|string>) 容错率更高.
      • 例如: 由于误操作, 可能多写几个分隔符, 将 /a/b/c 写成 /a//b///c, 此时通过 STREQUAL 来比较这两个路径得到的结果为不相等, 而通过 PATH_EQUAL 去比较两个路径得到的结果为相等.

    - 如果只给出了一个 loop_var, 那么它将使用一系列 loop_var_N 变量来存储对应列表中的当前项, 即 loop_var_0 对应第一个列表、loop_var_1 对应第二个列表, 以此类推.
    - 如果遍历的多个列表中一个列表较短, 当它遍历完成之后将不会再参与后续的遍历 (用空""代替).
    - IN: 关键字, 表示在 xxx 中取值.
    - ZIP_LISTS: 关键字, 对应的是列表 list, 通过set 、list 可以获得.
