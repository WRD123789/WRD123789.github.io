网络与网络编程

网络与网络编程

网络协议和网络模型

在使用网络通信时, 可以认为是由一端程序发起了请求而由另一端回复了一个响应. 发起请求的那一端称作客户端, 响应的那一端称作服务端. 在不同的网络应用中, 客户端和服务端的组织结构会有一定的变化, 有些应用可能会使用一个固定的服务端来响应大量的客户端请求, 有些应用则让程序既充当客户端也充当服务端.

在网络通信中, 网络协议规定了通信两端数据交互的规则.

由于现实世界的网络通信纷繁复杂, 要关注许多问题, 如硬件如何设计、数据如何选择传播方向、如何提供可靠的网络通信等, 网络设计时不得不将其分解为多个层次, 每个层次处理独立的任务. 各层次之间存在着一定关系, 上层依赖于下层, 它需要使用下层提供的接口才能工作.

在不同的规范体系中, 层次结构会有所不同, 其中 OSI 网络模型有七层: 应用 层、表示层、会话层、传输层、网络层、数据链路层和物理层, 而现在广泛使用的 TCP/IP 协议族网络模型则有四层: 应用层、传输层、网络层和网络接口层. 如下图所示:

TCP/IP 协议族

在 TCP/IP 模型中, 完成每一层功能的实体如下:

  • 网络接口层 - 物理层: 通常指实际传输的物理设备, 如线缆、调制解调器等设备, 数据在这些设备中以比特流的形式传播.
  • 网络接口层 - 数据链路层: 通常指网卡的驱动程序. 该层将比特流转换为数据帧 (frame), 而 MAC 地址用来描述不同数据链路层设备的地址.
  • 网络层: 通常指不同的主机. 网络层负责将数据帧封装成 IP 数据报. 使用 IP 地址来描述主机在网络层的位置. 网络层之间采用逐机跳转的通信方式.
  • 传输层: 负责将 IP 数据报封装成 TCP、UDP 报文. 该层可以为两台主机的应用程序提供端到端的通信. 传输层只关心通信的起始端和目的端, 不关心数据的中转过程.
  • 应用层: 负责处理应用程序的逻辑.

可见, 协议是在相同的层次上生效的. 每个层次能够处理的数据单元称作协议数据单元 (PDU), 不同层次之间的协议数据单元是不同的, 因此, 数据在上下层之间传递时, 需要使用一定的方式转换协议数据单元.

在网络中, 除了需要有能够完整处理 4 层协议的主机外, 还需要有一些只支持部分协议的设备, 如路由器只支持最高到网络层的协议处理, 它可以协助转发网络层的数据报到更远的网络中. 而交换机只工作在网络接口层, 它能让数据帧传播到更远的链路上.

协议数据单元的转换

当数据从上层转移到下层时, 下层的数据单元会包含上层数据单元的所有内容, 将其作为下层协议数据单元的有效载荷, 此外, 还需要在头部或者是尾部增加下层的控制信息. 其中 TCP 段/ UDP 报文会在数据中增加端口等信息作为头部, IP 数据报会在 TCP 段/ UDP 报文内容的头部添加 IP 地址等控制信息. 如下图所示:

在数据链路层存在一个 MTU 参数, 它的数值决定了 IP 报文段的最大长度 (字节), 长度超过 MTU 的 IP 报文段将无法传递, MTU 的值通常为 1500、576, 而 IP 数据报的总长度上限为 65535 字节. 因此, 当数据在网络层和数据链路层之间传递时, 网络层应用需要将 IP 数据报按照下层的 MTU 进行分片. 在到达目的主机后, 网络层应用还需要将各分片重组. 此外, 在成帧时, 除了需要增加一个链路头, 还需要在末尾添加 CRC 校验和, 来检测传输的过程中是否出现硬件错误导致的比特跳变.

虽然我们可以将每一层都看成是一个独立的进程, 但在实际的Linux系统中, 数据链路层、网络层和传输层的各种协议的相关代码已经由内核开发者实现, 并且作为内核的一部分运行, 这被称为内核协议栈. 此时, 网络应用开发者只需要关注应用层协议的处理, 通过调用内核协议栈提供的接口即可实现期望的功能.

本地环回设备

本地环回设备是一个用于网络测试和开发的虚拟网络接口. 它的主要功能是允许网络应用程序在不需要实际网络连接的情况下进行通信. 通常, 环回设备的 IP 地址是 127.0.0.1, 其主机名为 localhost. 它通常用于在同一台机器上进行网络通信, 通过环回接口发送的数据会立即返回给发送端 (只通过传输层和网络层). 并且, 环回设备的通信不会受到 MTU 的限制.

以太网

以太网与交换机

以太网是一种通过铜缆或光纤实现的有线网络技术, 用于连接局域网中的计算机和其他设备. 早期的以太网通常是共享的: 多个主机共享一个网络介质传播数据, 如果多个主机同时发送数据, 则需要采用一些算法来避免碰撞. 随着网络速度的提升, 共享以太网逐渐被一个星形拓扑接口的网络所取代: 每台主机通过一条独享的网线和交换机连接, 而交换机则负责为每台主机提供全双工通信. 如下图所示:

交换机在数据链路层工作, 通过 MAC 地址来转发数据帧. 它会维护一个 MAC 地址表, 记录每个端口连接的设备的 MAC 地址. 当交换机接收到一个数据帧时, 会检查帧的目的MAC地址, 并根据MAC地址表决定将帧转发到哪个端口. 如果目的MAC地址不在表中, 交换机会将帧广播到所有端口. 交换机会在收到帧时学习源 MAC 地址, 并将其添加到 MAC 地址表中, 以优化未来的帧转发.

    • 长度: 4 字节
    • 功能: 使用循环冗余校验 (CRC) 算法计算的校验码, 用于检测帧在传输过程中是否发生错误.

ARP 协议

在网络传输中, 我们使用 IP 地址来定位主机的位置, 但是 IP 地址是不固定的, 而网络设备在链路层中的 MAC 地址是固定的 (由生产厂商分配). 在构建以太网帧时, 网络层的 IP 地址不能够直接填入帧头中, 因此, 我们需要一种协议来将 IP 地址转换为 MAC 地址, 这便是 ARP 协议. ARP 协议可以将 32 位的 IPv4 地址解析为 48 位的 MAC 地址, 并且会随着时间的变化动态调整来匹配实际的网络拓扑结构.

ARP 是通过广播请求和单播回复来完成的, 主要步骤如下:

  1. ARP 请求:
    • 当主机 A 需要发送数据给主机 B, 但只知道主机 B 的 IP 地址, 不知道其 MAC 地址时, 主机 A 会在其子网内发送一个 ARP 请求广播帧.
    • 这个帧包含了主机 A 的 IP 地址和 MAC 地址, 以及主机 B 的 IP 地址. 请求的目的是询问网络中哪个主机拥有这个 IP 地址对应的 MAC 地址.
  2. ARP 回复:
    • 网络中所有主机接收到 ARP 请求后, 会检查请求中的目标 IP 地址是否与自己的 IP 地址匹配.
    • 如果匹配, 主机 B 会发送一个 ARP 回复帧给主机 A, 回复中包含了主机 B 的 MAC 地址.
    • 如果不匹配, 主机会丢弃该请求.
  3. 缓存:
    • 主机 A 收到 ARP 回复后, 会将主机 B 的 IP 地址和 MAC 地址映射关系缓存到本地主机的 ARP 缓存中, 以便后续通信中直接使用.
    • ARP 缓存也称为 ARP 表, 用于存储 IP 地址到 MAC 地址的映射关系. 主机在每次通信时会首先检查 ARP 表, 如果找不到对应的映射关系, 才会发送 ARP 请求. ARP 表中的条目有一定的生存时间, 过期后会被删除, 以确保网络地址的动态变化.

IP 协议

IP 协议是互联网的核心协议之一, 它提供了将数据包从源地址传输到目标地址的机制.

IP 协议将传输的数据分成一个个小的数据包 (称为数据报), 并为每个数据报添加头部信息. 头部信息包括源地址和目标地址、数据包的长度、标识、偏移、TTL (生存时间) 等. 然后, 这些数据包通过网络传输到目标地址.

IP 协议提供的数据报交付服务是不可靠的、无连接的. 不可靠意味着如果传递数据报的过程中出现差错, 网络层可以选择丢弃数据报, 而不会主动重传; 无连接意味着 IP 协议不会记录传递数据的路径, 因此, 两个主机之间传递的不同数据报可能会走不同的路径, 并且有可能不按传输顺序到达目的主机.

IP 数据报

正常的 IPv4 头部大小为 20 个字节 (如果不使用选项字段). 由于 TCP/IP 协议中所有协议的头部都采用大端法进行传输, 所以大端法也称作 “网络字节序”. 由于大多数 PC 使用的是小端法, 所以在构造完头部之后、在传输之前需要对其执行大小端转换.

其中各字段含义如下:

  • Version (版本): 4 位, 表示 IP 协议的版本号, IPv4 的版本号为 4.
  • IHL (头部长度): 4 位, 表示 IP 头部的长度, 以 32 位 (4 字节)为单位. 最小值为 5 (表示 20 字节), 最大值为 15 (表示 60 字节).
  • Type of Service (服务类型): 8 位, 用于描述数据包的优先级和服务质量.
  • Total Length (总长度): 16 位, 表示整个 IP 数据报 (包括头部和数据部分)的总长度, 以字节为单位. 最大值为 65535 字节.
  • Identification (标识): 16 位, 用于唯一标识主机发送的每一个数据包.
  • Flags (标志): 3 位, 包括未使用位、DF (不分片标志)、MF (更多分片标志).
  • Fragment Offset (片偏移): 13 位, 用于指示分片数据包在原始数据包中的位置.
  • Time to Live (生存时间): 8 位, 表示数据包可以经过的最大路由器数目, 防止数据包在网络中无限循环.
  • Protocol (协议): 8 位, 表示数据部分所使用的协议, 例如 TCP (6)、UDP (17).
  • Header Checksum (头部校验和): 16 位, 用于验证头部数据的完整性.
  • Source Address (源地址): 32 位, 表示发送数据包的设备的 IP 地址.
  • Destination Address (目标地址): 32 位, 表示接收数据包的设备的 IP 地址.
  • Options (选项): 可变长度, 可选字段, 提供额外的控制和调试信息.
  • Padding (填充): 填充到 32 位的整数倍.

分片与重组

每种数据链路的最大传输单元 MTU 都是不相同的, 如 FDDI 数据链路 MTU 4352、以太网的 MTU 是 1500 字节等. 常见的数据链路是以太网, 它的 MTU 是 1500 字节.

当 IP 数据报的大小大于 MTU 时, 就会被分片, 分片后的数据在到达目的主机之前不会被重组, 在目的主机的网络层中, 这些数据才会重组. 例如:

由于重组的过程比较复杂, 因此协议的设计者应该尽可能避免让 IP 数据包的大小大于 MTU. 并且, 在分片传输中，一旦某个分片丢失，就会造成整个 IP 数据报作废, 为此, TCP 引入了 MSS 也就是在 TCP 层进行分片不由 IP 层分片, 而对于 UDP 我们尽量不要发送一个大于 MTU 的数据报文.

路由器

链路层的数据是采用多跳的方式进行通信的, 也就是说一个主机只能决定数据报下一跳的目标, 而不能直接和最终的目的地建立连接. 在这种逐跳通信的网络中寻找起点和目的地之间一条合适通路的过程称为路由.

路由器是一种拥有两个以上接口的网络设备, 它们工作在网络层, 在经过路由决策后, 可以将数据从一个接口转发到另一个接口, 通常用于连接多个网络并在它们之间转发数据包. 因此, 路由器可以充当一个端到端通信系统的中间结点.

路由器的主要功能是 IP 转发, 它需要维护一个数据结构, 称为路由表, 表中的每一项通常包含如下字段:

  • 目标网络 (Destination Network): 目标网络或子网的 IP 地址和前缀长度. 例如, 192.168.1.0/24 表示一个目标子网.
  • 子网掩码 (Subnet Mask): 用于确定目标网络的网络部分和主机部分. 例如, 子网掩码 255.255.255.0 与目标网络 192.168.1.0 一起表示一个 /24 子网.
  • 下一跳 (Next Hop)：数据包应转发到的下一台设备的 IP 地址. 对于直接连接的网络, 下一跳可以是 on-link.
  • 接口 (Interface): 数据包应从路由器的哪个接口发出.
  • 度量值 (Metric): 用于衡量路径的优先级或成本, 度量值越低, 路径越优先. 度量值可以基于跳数、带宽、延迟等因素.
  • 路由类型 (Route Type): 表示路由条目的类型, 例如直接连接路由、静态路由或动态路由.

可以使用 route 指令查看主机的路由表、添加和删除路由表的表项:

NAT

网络地址转换 (NAT) 被设计来缓解 IPv4 地址不足的问题. NAT 用来在多个设备共享一个公共 IP 地址时管理和重写网络地址信息, 最常见的用途是在局域网中让多台设备通过一个公共 IP 地址访问互联网.

NAT 位于路由器或防火墙上, 当内网设备 (例如家庭网络中的电脑、手机等) 发出连接请求时, NAT 会将这些请求的源地址 (通常是私有 IP 地址, 如 192.168.1.x) 转换为公共 IP 地址. 相应地, 当返回数据包到达时, NAT 会将目的地址从公共 IP 转换回私有 IP 地址, 并将数据包转发到正确的内网设备.

NAT有几种主要类型:

  • 静态 NAT: 将一个固定的私有IP地址映射到一个固定的公共 IP 地址. 这种类型的 NAT 通常用于服务器等需要外部访问的设备.
  • 动态 NAT: 将私有 IP 地址池中的地址动态映射到公共 IP 地址池中的地址. 这种类型的 NAT 主要用于较大的网络中.
  • 端口地址转换: 也称为网络地址端口转换 (NAPT) 或端口重载. 它是动态 NAT 的一种形式, 它通过使用不同的端口号来区分多个私有 IP 地址, 这样可以让多个设备共享一个公共 IP 地址. 这种类型的 NAT 最常见于家庭网络和小型办公室网络. 例如:

TCP 协议

TCP (Transmission Control Protocol, 传输控制协议) 是面向连接的、可靠的、基于字节流的传输层通信协议, 提供可靠的、按顺序的和无差错的数据传输服务.

为了保证可靠性和流量控制所维护的状态信息 (如 Socket、序列号和窗口大小) 的组合称为连接. 因此, 建立一个 TCP 连接是需要客户端与服务端达成三个信息的共识:

  • Socket: 由 IP 地址和端口号组成.
  • 序列号: 用来解决乱序问题等
  • 窗口大小: 用来做流量控制

由源地址、源端口、目的地址、目的端口组成的四元组可以唯一地确定一个 TCP 连接. 其中源地址和目的地址的字段在 IP 头部, 源端口和目的端口的字段在 TCP 头部.

  • 序列号: 在建立连接时由计算机生成的随机数作为其初始值, 通过 SYN 包传给接收端主机, 每发送一次数据, 就累加一次该数据字节数的大小. 用来解决网络包乱序问题.
  • 确认应答号: 指下一次期望收到的数据的序列号, 发送端收到这个确认应答以后可以认为在这个序列号之前的数据都已经被正常接收. 用来解决丢包的问题.
  • 控制位:
    • ACK: 该位为 1 时, 确认应答的字段变为有效, TCP 规定除了最初建立连接时的 SYN 包之外该位必须设置为 1.
    • RST: 该位为 1 时, 表示 TCP 连接中出现异常必须强制断开连接.
    • SYN: 该位为 1 时, 表示希望建立连接, 并在其序列号字段进行序列号初始值的设定.
    • FIN: 该位为 1 时, 表示今后不会再有数据发送, 希望断开连接. 当通信结束希望断开连接时, 通信双方的主机之间就可以相互交换 FIN 位为 1 的 TCP 段.

连接断开

TCP 断开连接是通过四次挥手的方式来完成的, 双方都可以主动断开连接, 四次挥手的过程如下图所示:

整个过程详细介绍如下:

  1. 客户端打算关闭连接, 此时会发送一个 TCP 首部 FIN 标志位被置为 1 的报文, 即 FIN 报文, 之后客户端进入 FIN_WAIT_1 状态.
  2. 服务端收到该 FIN 报文后, 会向客户端发送 ACK 应答报文, 接着服务端进入 CLOSE_WAIT 状态.
  3. 客户端收到服务端的 ACK 应答报文后, 就进入 FIN_WAIT_2 状态.
  4. 服务端处理完数据后, 会向客户端发送 FIN 报文, 之后服务端进入 LAST_ACK 状态.
  5. 客户端收到服务端的 FIN 报文后, 会返回一个 ACK 应答报文, 之后进入 TIME_WAIT 状态.
  6. 服务端收到了 ACK 应答报文后, 就进入了 CLOSE 状态, 至此服务端已经完成连接的关闭.
  7. 客户端在经过 2MSL 一段时间后, 自动进入 CLOSE 状态, 至此客户端也完成连接的关闭.

另外, 只有主动关闭连接的一端, 才有 TIME_WAIT 状态.

基本特性

重传机制

当数据包丢失时, TCP 会采用重传机制解决.

常见的重传机制有:

  • 超时重传
  • 快速重传
  • SACK
  • D-SACK
超时重传

在发送数据时, 设定一个定时器, 当超过指定的时间后, 如果没有收到对方的 ACK 确认应答报文, 则重传该数据.

TCP 会在以下两种情况发生超时重传:

  • 数据包丢失
  • 确认应答丢失

在 Linux 中, 每当遇到一次超时重传时, 都会将下一次超时时间间隔设为先前值的两倍. 如果遇到连续两次超时, 就说明网络环境差, 不宜频繁发送.

显然, 超时触发重传存在的问题是, 超时周期可能相对较长.

快速重传不以时间为驱动, 而是以数据驱动重传.

虽然快速重传解决了超时时间过长的问题, 但是还面临着其他的问题: 重传的时候, 是只重传缺失的报文, 还是重传之后所有的报文. 例如, 假设发送方发送了 6 个数据, 编号的顺序是 Seq1 ~ Seq6, 但是 Seq2、Seq3 都丢失了, 那么当接收方收到 Seq4、Seq5、Seq6 时, 都会回复 ACK2 给发送方, 此时发送方并不清楚这连续的 ACK2 是接收方收到哪个报文而回复的, 此时接收方有两种选择:

  • 只选择重传 Seq2 一个报文. 此时重传的效率很低, 因为对于丢失的 Seq3 报文, 在后续收到三个重复的 ACK3 时才能触发重传.
  • 重传 Seq2 之后已发送的所有报文. 虽然能同时重传已丢失的 Seq2 和 Seq3 报文, 但是 Seq4、Seq5、Seq6 的报文已经被接收过了, 重复传输这些报文会浪费资源.

滑动窗口

滑动窗口是 TCP 中用于流量控制和数据传输管理的重要机制. 它确保了发送方和接收方能够高效地传输数据, 同时避免网络拥塞和数据丢失.

窗口的实现实际上是操作系统开辟的一个缓存空间, 发送方主机在等到确认应答返回之前, 必须在缓冲区中保留已发送的数据. 如果按期收到确认应答, 此时数据就可以从缓存区清除.

窗口大小是指无需等待确认应答, 而可以继续发送数据的最大值.

假设窗口大小为 3 个 TCP 段, 那么发送方就可以连续发送 3 个 TCP 段, 即使中途有 ACK 丢失, 也可以通过下一个确认应答进行确认. 如下图所示:

TCP 头部的 Window 字段表示窗口大小, 通过该字段, 接收端可以告诉发送端自己还有多少缓冲区可以接收数据. 通常窗口的大小是由接收方的窗口大小决定的, 发送方发送的数据大小不能超过接收方的窗口大小, 否则接收方就无法正常接收到数据.

拥塞控制

拥塞控制用于避免发送方的数据填满整个网络.

为了调节所要发送数据的量, 发送方需要维护一个名为拥塞窗口 cwnd 的状态变量, 它会根据网络的拥塞程度动态变化的. 此时, 发送窗口的大小为拥塞窗口和接收窗口的最小值.

拥塞窗口的变化规则是:

  • 只要网络中没有出现拥塞, cwnd 就会增大
  • 如果网络中出现了拥塞, cwnd 就减少

当发送方没有在规定时间内接收到 ACK 应答报文, 即发生超时重传时, 就会认为网络出现了拥塞.

主要通过四个算法实现拥塞控制:

  • 慢启动
  • 拥塞避免
  • 拥塞发生
  • 快速恢复
慢启动

TCP 在刚建立连接后, 首先有一个慢启动的过程: 发送方每收到一个 ACK, 拥塞窗口 cwnd 的大小就会加 1. 这个过程如下图所示:

  • 当 cwnd ≥\geq≥ ssthresh 时, 使用拥塞避免算法.
拥塞避免

拥塞避免算法的规则是: 每收到一个 ACK, cwnd 增加 1cwnd\frac{1}{cwnd}cwnd1​.

假设 ssthresh 为 8 (指 8 个 MSS 大小), 当 8 个 ACK 应答确认到来时, 每个确认使 cwnd 增加 18\frac{1}{8}81​, 此时, 发送方可以发送 9 个 MSS 大小的数据, 变为线性增长.

随着 cwnd 的增加, 网络会慢慢进入了拥塞状态, 此时就会出现丢包, 需要对丢失的数据包进行重传, 当触发了重传机制, 就进入了拥塞发生算法.

拥塞发生

重传机制主要有两种:

  • 超时重传
  • 快速重传

这两种机制使用的拥塞发送算法是不同的.

当发生超时重传时, 就会使用拥塞发生算法, 此时, ssthresh 和 cwnd 的值会发生变化:

  • ssthresh 会被设置为 cwnd2\frac{cwnd}{2}2cwnd​.
  • cwnd 重置为 1 (实际是恢复为 cwnd 的初始值).

整个过程如下图所示:

但是这种方式下 cwnd 的变化幅度过大, 另一种更好的方式是在发生快速重传时使用的拥塞发生算法, 此时 ssthresh 和 cwnd 的值变化如下:

  • cwnd = cwnd2\frac{cwnd}{2}2cwnd​.
  • ssthresh = cwnd.
  • 进入快速恢复算法.
快速恢复

快速恢复算法如下:

  • 拥塞窗口 cwnd = ssthresh + 3.
  • 重传丢失的数据包.
  • 如果再收到重复的 ACK, 那么 cwnd 增加 1.
  • 如果收到新数据的 ACK 后, 使 cwnd = ssthresh.

UDP 协议

UDP 是一种无连接协议, 不提供可靠的数据传输、数据顺序控制和流量控制等功能. 由于没有连接建立和维护的开销, UDP 适用于对传输速度要求高且对可靠性要求低的应用场景.
