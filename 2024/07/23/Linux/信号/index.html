信号

信号

在进程运行过程中, 经常会产生一些事件, 这些事件的产生和进程的执行往往是异步的, 信号提供了一种在软件层面上的异步事件处理机制. 由于在硬件层面上的异步处理机制是中断, 而信号是中断的软件模拟, 因此信号是一种软件中断.

每个信号都可以用一个整型常量宏表示, 以 SIG 开头, 如 SIGCHLD、SIGINT 等, 它们一般在系统头文件 signal.h 中定义, 可以通过 kill -l 指令查看信号列表.

信号的产生

产生的信号的方法有很多种:

  1. 终端输入: 用户可以在终端使用键盘组合键产生信号. 例如, 按下 Ctrl+C 会产生 SIGINT 信号, 按下 Ctrl+\ 会产生 SIGQUIT 信号.
  2. 系统调用: 进程可以通过系统调用向自己或其他进程发送信号. 例如:
    • kill(pid_t pid, int sig): 向指定进程发送信号.
    • raise(int sig): 向自身进程发送信号.
    • abort(): 向自身进程发送 SIGABRT 信号, 用于异常终止.
  3. 定时器: 进程可以使用定时器函数设置定时器, 当定时器到期时会发送信号. 例如:
    • alarm(unsigned int seconds): 在指定的秒数后发送 SIGALRM 信号.
    • setitimer 和 timer_create 等更高级的定时器函数.
  4. 硬件异常: 当进程执行非法操作 (如非法内存访问、除零操作等) 时, 内核会向进程发送相应的信号. 例如, 非法内存访问会产生 SIGSEGV 信号.
  5. 其他进程: 一个进程可以使用 kill 系统调用向另一个进程发送信号. 例如, kill(1234, SIGKILL) 向进程号为 1234 的进程发送 SIGKILL 信号.

信号的处理

进程接收到信号后的常见处理方式如下:

  1. 执行默认处理: 每种信号都有一个默认处理动作, 除非被显式覆盖. 例如, SIGINT 的默认处理是终止进程, SIGSEGV 的默认处理是产生核心转储并终止进程.
  2. 捕获信号: 进程可以通过 signal 或 sigaction 函数注册一个信号处理程序. 当信号到达时, 内核会调用该处理程序.
  3. 忽略信号: 进程可以通过 signal 函数设置信号处理程序为 SIG_IGN 来忽略某个信号. 例如, signal(SIGINT, SIG_IGN) 会忽略 SIGINT 信号.
  4. 阻塞信号: 进程可以使用 sigprocmask 函数阻塞某些信号, 从而暂时不处理这些信号. 阻塞的信号会在解除阻塞时处理.

但是有两个信号既不能被忽略也不能被捕捉, 它们是 SIGKILL 和 SIGSTOP, 如果进程接收到这两个信号, 只能接受系统的默认处理.

信号的实现机制

信号可以被视为一组标志位, 每个信号对应一个标志位. 进程的信号集 (sigset_t) 中每个位置对应一个特定的信号, 定义如下:

其中两个重要的信号集如下:

  • sigblock 集 (也称为信号屏蔽字). 该信号集包含所有应该被进程阻塞的信号. 当一个信号在 sigblock 集中时, 它不会立即传递给进程进行处理, 而是处于挂起状态.
  • sigpending 集 (也称为挂起信号集). 该信号集包含所有已发送给进程但尚未被处理的信号. sigpending 集中的信号表示已经被内核记录, 但由于该信号被阻塞或进程尚未准备好处理, 它们还没有被传递给进程.

信号的整个处理过程如下:

  1. 当信号产生时, 内核首先会检查目标进程的信号屏蔽字 (sigblock 集).
  2. 如果产生的信号在 sigblock 集中, 那么该信号会被阻塞. 阻塞的信号会被添加到 sigpending 集中, 表示该信号处于挂起状态, 这种信号被称为挂起信号或者是未决信号; 如果产生的信号不在 sigblock 集中, 目标进程会立刻处理该信号. 处理该信号时, 进程会暂停当前的任务, 执行信号处理程序 (如果有自定义处理程序) 或者执行默认处理行为.
    • 当进程解除对某个信号的阻塞时 (例如通过 sigprocmask), 该信号不会立即被处理, 而是需要等到下一个信号响应时机 (信号到达并可以被进程处理的时刻, 如从系统调用返回用户态、进程状态切换等).
  3. 在信号响应时机, 内核会检查进程的 sigpending 集, 查看是否有未决信号需要处理. 如果有未决信号, 内核会检查该信号是否被 sigblock 集阻塞. 如果未决信号不再被阻塞 (即不在 sigblock 集中), 内核会进行信号处理.

同步信号是由进程自身的行为所触发的信号. 这些信号与进程的执行流紧密相关, 通常在特定的代码执行过程中产生, 它们的触发在代码中是可以预测的; 异步信号是由外部事件触发的信号, 不一定与进程的当前执行流直接相关, 通常由外部资源或用户操作引起, 它们的触发是随机的, 无法在代码中预测到具体的触发时机.

在执行信号处理程序时, 如果接收到一个信号, 会根据该信号的类型作出不同的反应:

  • 如果接收到的是另一个不同类型的信号. 当前的信号处理程序会被中断, 此时 CPU 会优先执行新的信号的信号处理程序, 执行完成后, 再恢复原来的信号处理程序的上下文.
  • 如果接收到的是另一个相同类型的信号. 当前的信号处理程序不会被中断, 在执行完当前的信号处理程序后, 才会执行新的信号的信号处理程序.
  • 如果接收到多个重复的相同类型的信号, 后面的信号会被忽略.

当进程在执行某个信号处理函数时, 如果再产生一个同类型信号, 信号处理流程不会中断 (因为在执行该函数时, 信号集 sigblock 会将本信号对应位设置为 1), 而信号集 sigpending 中的对应位会被设置为 1, 表示此时存在一个挂起信号, 随后产生的同类型信号将不再被记录. 之所以设计同类型信号无法被中断, 是考虑到信号处理函数执行过程中可能会修改静态数据或者堆数据 (如 stdout), 如果中断该过程, 可能会破坏这些数据.

    • SA_RESTART: 自动重启被信号中断的系统调用.
    • SA_SIGINFO: 使用 sa_sigaction 处理信号, 而不是 sa_handler.

主动发送信号

kill 系统调用用于向进程发送信号. 该函数的原型如下:

参数:

  • pid: 这是一个进程 ID. kill 可以向不同类型的进程发送信号, 具体如下:
    • pid > 0: 发送信号 sig 给 ID 为 pid 的进程.
    • pid == 0: 发送信号 sig 给当前进程所在进程组中的所有进程.
    • pid < -1: 发送信号 sig 给进程组 ID 为 -pid 的所有进程.
    • pid == -1: 发送信号 sig 给系统中所有有权限接收此信号的进程 (除了 PID 为 1 的进程, 即 init 进程).
  • sig: 要发送的信号编号. 常见信号包括：
    • SIGKILL (9): 强制终止进程.
    • SIGTERM (15): 请求终止进程，进程可以捕获此信号并进行清理操作.
    • SIGSTOP (19): 暂停进程.
    • SIGCONT (18): 继续执行暂停的进程.

返回值

  • 成功时, 返回 0.
  • 失败时, 返回 -1, 并设置 errno 来指示具体的错误.

时钟处理

系统时钟是由操作系统管理的一个计时器, 通常是通过定时中断来更新的. Linux 内核使用系统时钟来跟踪当前时间和日期. 系统时钟通常由一个定时器中断源来驱动. 该定时器以固定频率生成中断, 内核在每次中断时更新系统时钟, 并处理与时间相关的任务, 如进程调度、维护 uptime、更新系统时间等.

硬件时钟, 也称为实时时钟, 它是主板上独立于系统时钟的硬件组件, 即使在系统关闭时也能保持运行. 启动时, 系统会读取硬件时钟的时间来设置系统时钟.

操作系统为每个进程维护三种不同的定时器, 分别是

  • 真实计时器: 基于实际经过的时间的计时器. 它不管进程的状态 (运行、等待或休眠), 只要设定的时间到了, 就会触发一个 SIGALRM 信号.
  • 虚拟计时器: 仅在进程运行用户态代码时计时. 如果进程处于等待状态或执行内核态代码, 则不会计时. 当设定的时间到达时, 系统发送 SIGVTALRM 信号给进程.
  • 实用计时器: 在进程运行用户态和内核态代码时都计时, 但不包括等待时间. 当设定的时间到达时, 系统发送 SIGPROF 信号给进程.
