文件 I/O: 通用的 I/O 模型

所有执行 I/O 操作的系统调用都以文件描述符, 一个非负整数, 来指代打开的文件. 文件描述符用来表示所有类型的已打开文件, 包括管道 (pipe)、FIFO、socket、终端、设备和普通文件. 每个进程的文件描述符相互独立.

大多数程序都期望能够使用 3 种标准的文件描述符:

文件描述符 用途 POSIX 名称 stdio 流
0 标准输入 STDIN_FILENO stdin
1 标准输出 STDOUT_FILENO stdout
2 标准错误 STDERR_FILENO stderr

在 shell 中, 这 3 个文件描述符始终是打开的, 当 shell 创建新的进程 (调用 fork 系统调用) 来执行程序时, 这三个文件描述符就被复制了 (调用 fork 创建子进程时, 会复制父进程的文件描述符).

在程序中指代上述文件描述符时, 可以使用数字 (如 0、1), 也可以使用 <unistd.h> 所定义的 POSIX 标准名称 (更合适).

  • 虽然 stdin、stdout 和 stderr 变量在程序初始化时用于指代进程的标准输入、标准输 出和标准错误, 但是调用 freopen() 库函数可以使这些变量指代其他任何文件对象. 也就是说, 针对 stdout 调用 freopen() 函数后, 无法保证 stdout 变量的值仍然为 1.

下面是执行文件 I/O 操作的 4 个主要系统调用 (编程语言和软件包通常会利用 I/O 函数库间接调用它们):

  • fd = open(pathname, flags, mode) 函数打开 pathname 所标识的文件, 并返回文件描述符, 用来在后续函数调用中指代打开的文件. 如果文件不存在, open() 函数可以创建它, 这取决于对位掩码参数 flags 的设置. flags 参数还可以指定文件的打开方式: 只 读、只写或读写方式. mode 参数则指定了由 open() 创建的文件的访问权限, 如果 open() 函数并未创建文件, 那么可以忽略 mode 参数.
  • numread = read(fd, buffer, count) 函数从 fd 所指代的打开文件中读取至多 count 字节的数据, 并存储到 buffer 中. read() 的返回值为实际读取到的字节数. 如果再无字节可读, 则返回值为 0.
  • numwritten = write(fd, buffer, count) 函数从 buffer 中读取至多 count 字节的数据写入由 fd 所指代的已打开文件中. write() 的返回值为实际写入文件中的字节数.
  • status = close(fd) 在所有输入/输出操作完成后, 调用 close(), 释放文件描述符 fd 以及与之相关的资源.

通用 I/O

UNIX I/O 模型的显著特点之一是其输入/输出的通用性概念. 这意味着使用 4 个系统调用 open()、read()、write() 和 close() 就可以对所有类型的文件执行 I/O 操作, 包括终端之类的设备. 因此, 仅使用这些系统调用编写的程序, 将对任何类型的文件有效.

要实现通用 I/O, 就必须确保每一文件系统和设备驱动程序都实现了相同的 I/O 系统调用集. 在编程时, 通常可以忽略设备专有的特性, 如果需要访问文件系统或设备的专有功能时, 可以选择 ioctl() 系统调用, 该调用为通用 I/O 模型之外的专有特性提供了访问接口.

打开文件: open()

open() 函数既能打开一个已存在的文件, 也能创建并打开一个新的的文件.

要打开的文件由参数 pathname 标识. 如果调用成功, open() 将返回一个文件描述符; 调用失败则返回 -1, 并将 errno 置为相应的错误标志.

参数 flags 是位掩码, 用于指定文件的访问模式和文件状态标志, 有以下几种文件访问模式:

访问模式 描述
O_RDONLY 以只读方式打开文件
O_WRONLY 以只写方式打开文件
O_RDWR 以读写方式打开文件

早期的 UNIX 实现中使用数字 0、1、2, 而非上表中所列的常量名称, 而大多数现代 UNIX 实现将这些常量定义为上述相应数字 (如 #define O_RDONLY 0), 来兼容早期的系统.

位掩码参数 mode 指定了文件的访问权限, 如果 open() 并未指定 O_CREAT 标志, 则可以省略 mode 参数. 需要注意的是, 新建文件的访问权限不仅仅依赖于参数 mode, 还受到进程的 umask 值和 (可能存在的) 父目录的默认访问控制列表的影响.

open() 函数中的 flags 参数

在 open() 函数中, flags 参数除了拥有文件访问模式标志外, 还有一些其他的操作标志, 如下表所示:

标志 用途
O_RDONLY 以只读方式打开
O_WRONLY 以只写方式打开
O_RDWR 以读写方式打开
O_CLOEXEC 设置 close-on-exec 标志
O_CREAT 若文件不存在则创建它
O_DIRECT 无缓冲的输入/输出
O_DIRECTORY 如果 pathname 不是目录, 则失败
O_EXCL 结合 O_CREAT 参数使用, 专门用于创建文件
O_LARGEFILE 在 32 位系统中使用此标志打开大文件
O_NOATIME 调用 read() 时, 不修改文件最近访问时间
O_NOCTTY 不让 pathname (所指向的终端设备) 成为控制终端
O_NOFOLLOW 不解引用符号链接
O_TRUNC 清空文件内容, 使其长度变为零
O_APPEND 总在文件尾部追加数据
O_ASYNC 当 I/O 操作可行时, 产生信号通知进程
O_DSYNC 提供了同步的 I/O 数据完整性
O_NONBLOCK 以非阻塞方式打开
O_SYNC 以同步方式写入文件

上述标志主要分为以下几组:

  • 文件访问模式标志, 即 O_RDONLY、O_WRONLY 和 O_RDWR. 调用 fcntl() 的 F_GETFL 操作能够检索文件的访问模式.
  • 文件创建标志. 这些标志不能检索, 也无法修改.
  • 文件状态标志. 可以使用 fcntl() 的 F_GETFL 和 F_SETFL 操作分别检索和修改此类标志.

在 Linux 中, 读取位于 /proc/PID/fdinfo 目录下的 linux 系统专有文件, 可以获取系统内任一进程中文件描述符的相关信息. 针对进程中每一个已打开的文件描述符, 该目录下都有相应的文件, 这些文件以对应的文件描述符的数值命名.
例如:

  • pos 表示当前的文件偏移量.
  • flags 是文件访问标志和已打开文件的状态标志.

下面是 flags 常量的详细描述:

  • O_APPEND. 在文件尾部追加数据.
  • O_ASYNC. 当可以在 open() 函数所返回的文件描述符实施 I/O 操作时, 系统会产生一个信号通知进程, 这仅对特定类型的文件有效, 如终端、管道、FIFO 和 socket. 在 Linux 中, 调用 open() 时指定 O_ASYNC 标志没有任何实质效果. 要启用这一特性, 必须调用 fcntl() 的 F_SETFL 来设置 O_ASYNC 标志.
  • O_CLOEXEC. 它会为新的文件描述符设置 close-on-exec 标志, 这会导致在任何 exec 系列函数执行成功时, 文件描述符自动关闭. 此时, 新的程序无法访问到这个文件描述符, 从而避免了可能的安全风险或错误.
  • O_CREAT. 如果文件不存在, 将创建一个新的空文件. 即使文件以只读方式打开, 此标志依然有效. 如果在 open() 函数中指定 O_CREAT 标志, 那么还需要提供 mode 参数, 否则, 会将新文件的权限设置为栈中某个随机数.
  • O_DIRECT. 无系统缓冲的文件 I/O 操作. 如果要使用 O_DIRECT, 你需要在包含 <fcntl.h> 头文件之前定义 _GNU_SOURCE 功能测试宏. 因为 O_DIRECT 是 GNU C 库的一个扩展特性, 而不是 POSIX 标准的一部分, 只有在定义了 _GNU_SOURCE 宏的情况下, O_DIRECT 才会在 <fcntl.h> 头文件中被定义.
  • O_DIRECTORY. 如果 pathname 参数并非目录, 将返回错误 (错误号 errno 为 ENOTDIR). 这一标志是专为实现 opendir() 函数而设计的扩展标志. 同样, 为了使 O_DIRECTORY 标志的定义在 <fcntl.h> 中有效, 必须定义 _GNU_SOURCE 功能测试宏.
  • O_DSYNC. 该标志提供了数据同步 I/O 的完整性, 这意味着写操作会将数据刷新到底层硬件. 它不仅能保证文件数据已写入磁盘, 还确保更新后的文件元数据也已写入磁盘.
  • O_EXCL. 此标志与 O_CREAT 标志结合使用时, 如果文件已经存在, 就不会打开文件, 并且 open() 会调用失败, 返回错误, 设置错误号 errno 为 EEXIST. 其中检查文件存在与否和创建文件这两步属于同一原子操作. 这通常用于避免文件被意外覆盖, 也能防止文件被多个进程同时创建和修改. 另外, 如果在 flags 参数中同时指定了 O_CREAT 和 O_EXCL 标志, 且 pathname 参数是符号链接, 则 open() 函数调用会失败, 这样设计是要求有特权的应用程序在已知目录下创建文件.
  • O_LARGEFILE. 在 32 位操作系统中使用此标志, 以支持大文件操作. 此标志在诸如 Alpha、 IA-64 之类的 64 位 Linux 实现中是无效的.
  • O_NOATIME. 在读文件时, 不更新文件的最近访问时间. 要使用该标志, 要么调用进程的有效用户 ID 与文件的拥有者相匹配, 要么进程需要拥有特权 (CAP_FOWNER). 否则, open() 调用失败, 并返回错误, 错误号 errno 为 EPERM. 此标志是 Linux 特有的非标准扩展, 想要从 fcntl.h> 中启用此标志, 必须定义 _GNU_SOURCE 功能测试宏. O_NOATIME 标志旨在为索引和备份程序服务减少开销, 因为更新访问时间可能会带来额外的性能开销, 而缓存文件、日志文件等文件虽然需要频繁读数据, 但是不需要记录文件的访问时间.
  • O_NOCTTY. 如果正在打开的文件属于终端设备, O_NOCTTY 标志防止其成为控制终端. 如果正在打开的文件不是终端设备, 则此标志无效.
  • O_NOFOLLOW. 通常来说, 如果 pathname 参数是符号链接, open() 函数将对 pathname 参数进行解引用. 一旦在 open() 函数中指定了 O_NOFOLLOW 标志, 且 pathname 参数属于符号链接, 则 open() 函数将返回失败 (错误号 errno 为 ELOOP). 此标志在特权程序中极为有用, 能够确保 open() 函数不对符号链接进行解引用. 为使 O_NOFOLLOW 标志在 <fcntl.h> 中有效, 必须定义 _GNU_SOURCE 功能测试宏.
  • O_NONBLOCK. 以非阻塞方式打开文件.
  • O_SYNC. 以同步 I/O 方式打开文件.
  • O_TRUNC. 如果文件已经存在且为普通文件, 那么将清空文件内容, 将其长度置 0. 在 Linux 下使用此标志, 无论以读、写方式打开文件, 都可清空文件内容 (在这两种情况下, 都必须拥有对文件的写权限).

open() 函数的错误

若打开文件时发生错误, open() 函数将返回 −1, 错误号 errno 标识错误原因. 下面是一些可能发生的错误:

  • EACCES. 文件权限不允许调用进程以 flags 参数指定的方式打开文件.
  • EISDIR. 所指定的文件属于目录, 而调用者企图打开该文件进行写操作.
  • EMFILE. 进程已打开的文件描述符数量达到了进程资源限制所设定的上限.
  • ENFILE. 文件打开数量已经达到系统允许的上限.
  • ENOENT. 要么文件不存在且未指定 O_CREAT 标志, 要么指定了 O_CREAT 标志, 但 pathname 参数所指定路径的目录之一不存在, 或者 pathname 参数为符号链接, 而该链接指向的文件不存在.
  • EROFS. 所指定的文件属于只读文件系统, 而调用者试图以写的方式打开文件.
  • ETXTBSY. 所指定的文件为可执行文件, 且正在运行. 系统不允许修改正在运行的程序.

改变文件偏移量: lseek()

对于每个打开的文件, 系统内核会记录它的文件偏移量 (也称为读写偏移量或指针). 文件偏移量是指执行下一个 read() 或 write() 操作时文件的起始位置.

当文件打开时, 会将文件偏移量设置为 0, 即指向文件的起始位置, 之后的每次 read() 或 write() 调用的执行将自动对其进行调整, 以指向已读或已写数据的下一字节.

针对文件描述符 fd 参数所指代的已打开文件, lseek() 系统调用根据 offset 和 whence 参数值调整该文件的文件偏移量:

其中 offset 参数是一个以字节为单位的数值. whence 参数则表明应参照哪个基点来解释 offset 参数, 该参数的可选值如下:

  • SEEK_SET. 将文件偏移量设置为从文件起始点开始的 offset 个字节.
  • SEEK_CUR. 将文件偏移量设置为从当前文件偏移量开始的 offset 个字节.
  • SEEK_END. 将文件偏移量设置为从当前文件末尾 (文件最后一个字节的下一个字节) 开始的 offset 个字节.

上述可选值可以以下图表示:

如果 whence 参数值为 SEEK_CUR 或 SEEK_END, offset 参数可以为正数也可以为负数；; 如果 whence 参数值为 SEEK_SET, offset 参数值必须为非负数.

lseek() 调用成功会返回新的文件偏移量. 该调用只是调整内核中与文件描述符相关的文件偏移量记录, 并没有访问物理设备.

lseek() 并不适用于所有类型的文件. 不允许将 lseek() 应用于管道、FIFO、socket 或者终端. 一旦如此, 调用将会失败, 并将 errno 置为 ESPIPE.

文件空洞

在程序的文件偏移量跨越文件结尾后, 再执行 I/O 操作, read() 调用将返回 0, 而 write() 调用仍然可以写入数据.

文件结尾之后到新写入数据之间的这块空间被称为文件空洞. 文件空洞通常以 0 填充, 但是不占用实际的磁盘空间 (因为没有实际的数据块与之对应). 一个文件的逻辑大小 (文件读取时的大小) 通常包含文件空洞的大小, 而该文件的实际大小 (实际占用磁盘空间的大小) 则不包含文件空洞的大小.

  • 由于文件系统以块为单位分配空间, 如果空洞的边界落在块内, 而非恰好落在块边界上, 则会分配一个完整的块来存储数据, 块中与空洞相关的部分则以空字节填充. 这是因为当文件中的空洞不对齐块的边界时，一个块内可能既包含实际数据, 也包含空洞, 而文件系统不能仅为块的一部分分配空间, 必须为整个块分配空间.
