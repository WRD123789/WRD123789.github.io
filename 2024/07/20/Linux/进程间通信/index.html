进程间通信

进程间通信

在多个进程之间, 如果需要进行通信的话, 隔离特性会造成一些通信的障碍. 所以我们需要一些手段来跨越隔离, 实现进程间通信 (InterProcess Communication, IPC)

管道

一种最自然的通信方式就是依赖文件系统, 一个进程打开文件并写入数据, 另一个进程则打开文件来读取数据, 但这种通信方式依赖于磁盘文件. 为了提高效率, 可以使用有名管道 (FIFO), 有名管道是文件系统中一种专门用于读写的文件, 它通过内核的管道缓冲区来传输数据.

如果两个进程直接存在亲缘关系, 则可以使用匿名管道 (也称作管道), 它不需要在文件系统中创建单独的文件, 而是在进程的执行过程中动态地创建和销毁.

共享内存

在进程的数量比较少的时候, 使用管道进行通信是比较自然的思路. 但是随着进程数量增加, pipe 的使用次数将会急剧增加. 除此以外, 使用管道通信的时候, 数据要从写端拷贝到内核管道缓冲区, 再从缓冲区拷贝到读端, 总共要进行两次拷贝, 性能较差. 为了提升进程间通信的效率, 可以使用共享内存.

对于进程而言, 代码中所使用的地址都是虚拟地址, 所操作的地址空间是虚拟地址空间. 当进程执行的时候, 如果发生访问内存的操作, 操作系统和硬件需要保证虚拟地址能够映射到物理内存上, 和这种地址转换相关的设备被称为内存管理单元 (MMU). 因此, 如果两个不同的进程使用相同的虚拟地址, 它们所对应的物理内存地址是不一样的. 共享内存就允许两个或者多个进程共享一个给定的物理存储区域. 当然为了实现共享, 内核会专门维护一个用来存储共享内存信息的数据结构.

参数:

虚拟地址和物理地址转换机制

为了实现虚拟内存机制, 操作系统需要和硬件 MMU 配合来实现虚拟地址到物理地址的映射.

地址的分类

在 x86 架构中, 地址通常被分为三类:

  • 逻辑地址: 逻辑地址是程序员在编写代码时使用的地址, 由段选择子和段内偏移组成.
  • 线性地址: 也称为虚拟地址, 是通过分段机制从逻辑地址转换得到的统一地址空间. 线性地址空间是由操作系统和 CPU 管理的一个虚拟化的地址空间, 可以比实际的物理内存大.
  • 物理地址: 物理地址是实际在计算机物理内存 (RAM) 中的地址. 线性地址通过分页机制映射到物理地址. 分页机制使用页表来管理线性地址到物理地址的映射.

分段是一种将线性地址空间划分为更小的受保护地址空间 (称为段) 的机制.

从上图可知, 要想将逻辑地址转换为线性地址, 我们需要:

  • 段选择器: 它是段寄存器中的地址, 用于选择一个段描述符.
  • 偏移量: 从段描述符中获取基地址, 并加上该偏移量, 可以计算出线性地址.
  • 全局描述符表: 该表的基地址 (线性地址) 存储在 GDTR 寄存器中, 该表包含了一系列段描述符.

段选择器

段选择器是存储在段寄存器中的一个 16 位的值, 用于从两个表中选择一个段描述符:

  • GDT, 全局描述符表, 用于全系统.
  • LDT, 局部描述符表, 用于每个进程, 在内核切换进程上下文时进行切换.

有六个段寄存器用于存储段选择器:

  • CS: 代码段寄存器
  • SS: 堆栈段寄存器
  • DS: 数据段寄存器
  • ES、FS、GS: 附加段寄存器

想要执行一个程序, 至少需要在 CS、SS 和 DS 段寄存器中装入有效的段选择器.

64 位处理器中的分段机制

在 64 位模式下, 分段机制通常被禁用, 来创建一个扁平的 64 位线性地址空间 (此时逻辑地址等于线性地址). 处理器将 CS、DS、ES、SS 段的基地址视为零, 从而创建一个等于逻辑地址的线性地址. FS 和 GS 段是例外. 这些段寄存器可以用作线性地址计算中的附加基址寄存器, 它们有助于地址局部数据和某些操作系统数据结构.

在早期计算机体系结构中, 分段机制用于解决内存小、程序和数据需要灵活管理的问题, 然而, 随着计算机硬件的发展, 内存容量大幅增加, 操作系统和应用程序的需求也发生了变化. 在禁用分段后, 内存地址空间变成了一个连续的 64 位线性地址空间, 这一设计能够简化内存管理, 避免了在分段之间进行地址转换的复杂性; 同时由于不需要额外的分段转换步骤, 也能够减少处理器的开销, 提高了内存访问速度; 此外, 禁用分段意味着不需要进行段限检查, 减少了可能出现的异常情况, 提高了系统的稳定性和性能. 唯一保留的 FS 和 GS 段寄存器提供了一些灵活性, 用于特定的内存访问需求.

硬件分页机制

Linux 中的硬件分页机制是虚拟内存管理的一部分, 用于将虚拟地址映射到物理地址. 分页机制的主要目的是为了提供一种抽象, 使程序可以使用比物理内存更大的地址空间, 同时也可以提高内存的使用效率和安全性.

在该机制中, 虚拟地址空间和物理内存被划分成固定的大小. 虚拟内存的基本单位是页, 通常页的大小为 4KB; 物理内存的基本单位是物理页 (也称为页框), 大小和虚拟内存的页相同. 在内核访问内存时, 每次访问都是以页为单位的.

Linux 的页管理

在 Linux 中, 每个用户进程都拥有自己独立的页目录表和页表集合, 这样不同进程的相同虚拟地址才可以映射到不同的物理地址.

用户进程的进程控制块 (在 Linux 中, 由 task_struct 结构体表示) 中的 mm 和 active_mm 成员与进程的内存管理相关. 这两个成员是指向进程地址空间的指针:

  • mm 成员. mm 成员是指向进程的内存描述符结构体 mm_struct 的指针. 该结构体用于描述该进程的地址空间, 其中包含了与进程地址空间相关的信息, 如:
    • 所有已经分配的虚拟地址集合, 使用链表和红黑树来管理.
    • 页目录表, 总共使用的页表的个数.
    • 代码段、数据段、堆、栈、环境变量、命令行参数的起止地址.
    • 总共使用的物理页数量, 栈、共享内存、文件映射使用的页数量等.
  • active_mm 成员. 也是指向 mm_struct 结构体的指针, 它用于指向当前正在使用的内存描述符. 该成员在处理内核线程时非常重要: 内核线程没有自己的独立地址空间, 因此它们的 mm 成员为 NULL. 当内核线程执行时, 它会使用最近一个用户进程的地址空间 (即 active_mm) 来进行内存操作.

信号量

信号量是一种用于进程间同步和控制资源访问的机制. 它可以有效地解决多个进程或线程之间的同步问题, 防止竞争条件和死锁等问题.

信号量是一个整型变量, 初始值一般为正数, 用来表示可以被访问的资源数量. 信号量有两种主要操作:

  1. P 操作: 如果信号量的值大于 0, 则将其减 1; 如果信号量的值为 0, 则进程进入等待状态, 直到信号量的值大于 0.
  2. V 操作: 将信号量的值加 1, 如果有进程在等待信号量, 则唤醒一个等待进程.

信号量值的检查和加减操作是原子操作 (也称为原语), 来保证信号量不受进程切换的影响. 如果信号量的值初始为 1, 则可以被称为二元信号量.

下面主要介绍 System V 信号量. System V 信号量实际上是一个信号量集合 (创建后, 会被分配在内核区), 可以包含多个信号量, 用来管理复杂的共享资源情况, 例如多个资源的同步访问.

System V 信号量的使用主要涉及以下几个操作:

  • 创建和获取信号量集合: semget 函数用于创建一个信号量集合或获取一个已有的信号量集合.
  • 操作信号量: semop 函数用于执行 P 操作和 V 操作, 即减小和增加信号量的值.
  • 控制信号量: semctl 函数用于对信号量集合进行控制操作, 包括初始化信号量、删除信号量等.

    • IPC_EXCL: 与 IPC_CREAT 一起使用, 确保如果信号量集合已存在, 则返回错误.
    • 权限标志, 如 0666 (表示读写权限).

返回值:

  • 成功时返回信号量的标识符.
  • 失败时返回 -1, 并设置 errno.

控制信号量集合

使用 semctl 函数用于控制信号量集合, 例如初始化信号量、获取信号量值、删除信号量集合等.

参数:

  • semid: 信号量集合的标识符.
  • semnum: 信号量在集合中的索引.
  • cmd: 控制命令, 如:
    • IPC_STAT: 获取信号量的状态.
    • IPC_SET: 设置信号量的状态 (能设置的值比较有限).
    • IPC_RMID: 删除信号量集合.
    • SETVAL: 设置单个信号量的值.
    • GETVAL: 获取单个信号量的值.
    • SETALL: 设置所有信号量的值 (传入该值时不必关注 semnum, 且传入的数组中元素的数量必须和信号量的数量相同).
    • GETALL: 获取所有信号量的值 (传入该值时不必关注 semnum, 且传入的数组中元素的数量必须和信号量的数量相同).
  • 变长参数 (...): 根据命令不同而不同, 例如 SETVAL 需要 union semun 作为参数.

返回值:

  • 成功时返回命令的执行结果 (如 IPC_SET 会返回 0, GETVAL 会返回信号量的值).
  • 失败时返回 -1, 并设置 errno.

信号量的性能

虽然信号量能够保证系统的正确性和防止资源竞争, 但是也会对系统性能产生一些影响, 主要原因如下:

  • 信号量的使用会导致上下文切换. 当一个进程因等待信号量而阻塞时, 操作系统会切换到另一个进程. 这种上下文切换是有开销的, 包括保存和恢复进程状态、切换页表等.
  • 信号量操作通常涉及系统调用, 例如 semop、semctl 等. 这些系统调用需要在用户态和内核态之间切换, 并执行相应的内核代码. 系统调用的频繁使用会增加系统开销, 降低性能.

消息队列

消息队列是一种进程间通信机制, 它维护了一个消息的链表, 允许进程通过发送和接收消息来通信. 消息队列与其他 IPC 机制相比, 提供了一种更为灵活和可靠的通信方式.

创建/获取消息队列

可以使用 msgget 创建一个新的消息队列或获取一个现有的消息队列标识符. 该函数的原型为:

参数:

  • key: 一个键值, 用于标识唯一的消息队列. 可以通过 ftok 函数生成这个键值.
  • msgflg: 消息队列的标志. 可以是以下值的组合:
    • 权限标志 (类似于文件权限, 如 0666 表示所有用户可读写).
    • 控制标志, 如 IPC_CREAT (如果不存在则创建) 和 IPC_EXCL (与 IPC_CREAT 一起使用, 如果消息队列已存在则返回错误).

返回值:

  • 成功时返回消息队列标识符 (非负整数).
  • 失败时返回 -1, 并设置 errno 以指示错误原因.

发送消息

可以使用 msgsnd 向消息队列发送消息. 该函数的原型如下:

参数:

  • msgp: 指向消息结构的指针. 消息结构必须包含一个 long 类型的成员 (消息类型) 和消息正文, 例如:

  • 失败时返回 -1, 并设置 errno 以指示错误原因.

接收消息

可以使用 msgrcv 从消息队列接收消息. 该函数的原型如下:

参数:

  • msqid: 消息队列标识符.
  • msgp: 指向消息结构的指针, 用于存放接收的消息. 消息结构必须包含一个 long 类型的成员 (消息类型) 和消息正文, 例如:

/proc 目录

/proc 文件系统是一种虚拟文件系统, 包含内核和进程的信息, 它为用户和应用程序提供了一个接口来访问内核和系统进程的内部信息. 它不是一个真实存在于磁盘上的文件系统, 而是由内核在运行时动态生成的 (存放在内存中).

/proc 中的文件常被称作虚拟文件, 为了查看及使用上的方便, 这些文件通常会按照相关性进行分类存储于不同的目录甚至子目录中, 基本上可以分为以下几类:

  • 内核和系统信息:
    • /proc/cpuinfo: 显示 CPU 信息.
    • /proc/meminfo: 显示内存使用情况.
    • /proc/version: 显示内核版本信息.
    • /proc/uptime: 显示系统运行时间和空闲时间.
    • /proc/loadavg: 显示系统的负载平均值.
  • 进程信息: /proc/[pid]/, 每个运行的进程都有一个对应的目录, 其中 [pid] 是进程 ID. 这个目录包含了与该进程相关的各种信息和状态文件, 例如:
    • /proc/[pid]/cmdline: 进程的启动命令行.
    • /proc/[pid]/cwd: 进程的当前工作目录.
    • /proc/[pid]/exe: 进程的可执行文件.
    • /proc/[pid]/fd/: 进程的文件描述符.
  • 系统参数和配置: /proc/sys/, 包含可调节的内核参数, 可以通过 sysctl 命令或直接写入这些文件来调整系统行为. 例如:
    • /proc/sys/net/ipv4/ip_forward: 控制 IP 转发.
    • /proc/sys/vm/swappiness: 控制交换空间的使用.
  • 诊断和调试:
    • /proc/kmsg: 内核消息.
    • /proc/modules: 当前加载的内核模块.
    • /proc/interrupts: 中断处理信息.
