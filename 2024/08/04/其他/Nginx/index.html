Nginx

Nginx

Nginx 是一个轻量级的 HTTP 服务器, 采用事件驱动的、异步非阻塞处理方式的框架, 通常用于服务端的反向代理和负载均衡, 支持高并发.

基本概念

正向代理

正向代理是一个位于客户端和目标服务器之间的服务器, 客户端通过正向代理访问目标服务器.

当客户端想要访问目标服务器时, 首先会向正向代理服务器发送请求, 说明要访问的目标服务器, 正向代理服务器接收到请求后, 代替客户端向目标服务器发送请求, 目标服务器会将响应返回给正向代理服务器, 再由正向代理服务器转发给客户端.

正向代理主要服务于客户端, 能够隐藏客户端信息.

反向代理

反向代理是一个位于客户端和目标服务器之间的服务器, 但它是代表目标服务器来处理客户端请求.

当客户端向目标服务器发出请求时, 实际上由反向代理服务器接收请求, 反向代理服务器接收到请求后, 会选择一台实际的目标服务器来处理请求, 目标服务器会将响应返回给反向代理服务器, 再由反向服务器转发给客户端.

反向代理主要服务于服务器, 能够隐藏服务器信息, 并且能够将客户端请求分配给多台目标服务器, 均衡负载, 提高可用性和性能.

负载均衡

负载均衡提供了一组策略来将请求从代理服务器上分发到真实的服务器上.

在 Nginx 中, 一共提供了三种负载均衡策略:

  • 轮询 (默认方式): 每个请求按时间顺序逐一分配到不同的后端服务器上.
  • 权重: weight 代表权重, 默认为 1, 权重越高被分配的客户端请求越多.
  • ip_hash: 每个请求根据访问 IP 的 Hash 结果分配, 这样每个访客可以固定地访问一个后端服务器.

复杂均衡能够分摊服务器集群压力, 保证客户端访问的稳定性.

动静分离

动静分离机制的核心思想是将静态资源 (如HTML、CSS、JavaScript、图像、视频等) 和动态内容 (如需要经过服务器端处理的PHP、Python、Java等脚本生成的内容) 分开处理, 分别由不同的服务器或不同的服务器模块来处理. 这样负责处理动态请求的服务器可以专注于动态请求的处理上, 从而提高效率, 同时, 也可以针对处理静态请求的服务器做优化, 提高静态请求的响应速度.

Nginx 可以用来处理静态资源, 同时将动态请求反向代理到后端的应用服务器 (如 Tomcat).

Nginx 设计理念

Nginx 整体架构

Nginx 的整体架构如下图所示:

Nginx 是一种多进程架构, 其中:

  • Master 进程主要负责管理和控制 Worker 进程; 加载和解析 Nginx 的配置文件, 管理全局配置; 接收和处理外部信号 (如启动、停止、重载配置等), 并将信号传递给 Worker 进程.
  • Worker 进程是 Nginx 处理客户端请求的核心, 每个 Worker 进程都是独立的, 负责处理具体的网络连接和请求.
  • Cache loader 和 Cache manager 进程用于处理缓存事宜.

上面这些进程都是单线程的, 进程之间通过共享内存通信, 其中 Master 进程以 root 的身份启动, 其他进程则是以非特权用户的身份启动.

Worker 进程工作流程

Nginx 的代码是高度模块化的, Worker 进程的代码由核心模块和多个功能模块组成. 核心模块的主要功能是维持一个事件循环, 并根据请求的处理阶段分配合适的功能模块来处理. 功能模块负责实现各种表现和应用层的功能, 例如读写网络和存储系统、转换文本内容、过滤输出、运行服务器端代码、反向代理请求到上游服务器等. 这种模块化架构允许开发者在不修改 Nginx 核心代码的情况下扩展 Web 服务器的功能.

为了高效处理大量连接、请求、网络管理和内容检索等操作, Nginx 使用了事件通知机制 (如 epoll), 以达到最佳性能.

需要特别注意的是, Nginx 的监听 socket 是由主进程创建的, 并通过 fork 系统调用让多个 Worker 进程共享. Worker 进程从共享的监听 socket 中 accept 新连接, 每个 Worker 进程内部执行高效的事件循环, 可以管理数千个连接. 如果同时运行多个 Worker 进程, 一个连接会分配给哪个 Worker 由操作系统内核决定, 不受 Nginx 控制.

Nginx 中 Worker 进程的工作流程如下:

  1. 当 Nginx 启动时, Master 进程读取配置文件, 并根据配置启动多个 Worker 进程.
  2. 每个 Worker 进程初始化必要的资源, 如打开日志文件.
  3. Worker 进程进入事件循环, 使用 epoll (Linux) 或 kqueue (FreeBSD、macOS) 等高效的 I/O 多路复用机制监听事件.
  4. 事件循环中, Worker 进程会等待文件描述符上的事件, 如新的连接请求、读写事件等. 虽然 Master 进程打开了监听套接字并通过 fork 传递给 Worker 进程, 但实际的连接建立和处理已经建立的连接上的事件都是由 Worker 进程完成的.
  5. 当有新的连接请求到来时, Worker 进程接受连接, 并将其添加到事件循环中.
  6. Worker 进程会根据配置和请求类型处理 HTTP 请求, 例如读取请求头、请求体, 处理 URL 重写、负载均衡、反向代理等.
  7. 当请求处理完成后, Worker 进程会发送响应给客户端.

Nginx 配置文件

Nginx 的配置文件是一个用于定义服务器行为和参数的文件. 它使用特定的语法来配置各种模块、虚拟主机、路由、负载均衡、缓存等.

Nginx 的配置文件采用集中式的文件组织方式, 存在一个主配置文件, 通常位于 /etc/nginx 或 /usr/local/etc/nginx, 命名为 nginx.conf, 而其他的配置属性可以放置在其他文件中, 在主配置文件中使用 include 指令可以将它们组合起来.

配置文件是由 Master 进程读取和处理的, 在编译后会以只读文件的形式提供给 Worker 进程.

使用场景

在目前的市场中, 最常见的 Nginx 的使用场景是充当企业服务端的边缘节点. 比如一些企业网站的外部入口由 Nginx 提供. 在用户请求的资源中, 有相当一部分内容是静态的, 这些内容可能在一段时间内不会发生改变, Nginx 能够高效地提供这些静态资源, 而对于其中的动态请求, 由于这些请求十分复杂、迭代速度极快, 所以 Nginx 不适合处理这些请求, 此时就需要使用重量级服务器 (并发量不高, 能处理复杂的业务), 比如 Apache 或 tomcat, 但是直接将这些产品暴露在外网会由于并发量过大而难以负载, 所以一种解决方案是使用反向代理, 客户端的请求会先经过 Nginx, 将高并发的压力转移到 Nginx 上, 再由 Nginx 将这些请求分配给多个业务服务端.

时至今日, Nginx 的功能又得到进一步泛化, 在分布式系统构建的服务端体系中, 这些边缘节点可以完成更多的工作, 从而进化成一个专门的产品, 称为API 网关, API网关作为分布式服务端的边缘节点, 支持以下功能:

  • 路由转发, 为客户端的请求选择合适的服务端.
  • 请求鉴权, 验证客户端的权限.
  • 流量控制和负载均衡, 限制接入的流量, 合理地转发流量.
  • 协议转换, 在外部转入的 HTTP 协议请求和内网的私有协议进行相互转换.

Nginx 配置

Nginx 的配置文件是用于定义和管理 Nginx 行为和功能的文件. 它使用一种特定的语法来配置服务器、虚拟主机、路由、负载均衡、缓存等功能.

Nginx 的主配置文件通常是 /etc/nginx/nginx.conf, 配置文件使用层次化的块结构来组织指令. 这些块包括全局块、events 块、http 块, 以及其他可能的块 (如 mail 块、stream 块等).

Nginx 的配置文件有以下特性:

  • 使用 { } 来描述作用域.
  • 使用 # 单行注释.
  • 配置指令用分号结尾, 多个参数用空格分隔, 如果参数本身有空格, 可以使用引号.
  • 使用单引号和双引号界定字符串, 使用反斜杠转义.
  • include 指令可以包含其他配置文件, 支持通配符 *.
  • 通过 $var 引用变量.

location 块通常出现在 server 块内, 通过定义 location 块, 可以为特定的 URL 路径配置不同的处理规则.

location 块的基本格式如下:

其中, modifier 用于指定匹配类型, 可以是以下几种:

  • =: 精确匹配.
  • ^~: 前缀匹配, 如果匹配成功, 不进行后续的正则表达式匹配.
  • ~: 区分大小写的正则表达式匹配.
  • ~*: 不区分大小写的正则表达式匹配.
  • @: 命名位置, 通常用于内部重定向.

location 块的匹配原则是最左匹配 (也称为最长前缀匹配), 也就是说 location /foo { ... } 能匹配以 /foo 开头的 URI, 如 http://example.com/foo/a.txt, 而无法匹配 http://example.com/bar/foo/a.txt.

Nginx 处理请求时, 会按照以下顺序检查 location 块:

  1. 精确匹配 (modifier 为 =): Nginx 首先检查是否有与请求 URI 完全匹配的 location 块.
  2. 最长前缀匹配 (modifier 为 ^~): 如果没有找到精确匹配, Nginx 会查找 ^~ 修饰符的最长前缀匹配. 这个块的优先级高于正则表达式匹配.
  3. 正则表达式匹配 (modifier 为 ~ 或 ~*): Nginx 按照配置文件中的顺序, 依次检查带 ~ 或 ~* 修饰符的正则表达式匹配块.
  4. 通用最长前缀匹配 (没有 modifier): 如果上述三种方式都没有找到匹配, Nginx 会选择最匹配的前缀. 在这种情况下, 如果有多个匹配的前缀, Nginx 会选择最长的一个.

一旦找到匹配的 location 块, Nginx 将使用该块中的配置处理请求.

Nginx 惊群问题

惊群现象就是当多个进程或线程在阻塞等待同一个事件时, 如果该事件发生, 就会唤醒所有等待的进程/线程, 但最终只可能有一个进程/线程能对该事件进行处理, 其他进程/线程会在争夺失败后重新休眠, 唤醒这些进程/线程会导致大量的上下文切换, 造成系统资源的浪费.

accept 惊群

当多个进程/线程调用 accept 函数在同一个 socket 上等待连接时, 一个新连接的到来就会导致所有阻塞在该 socket 上的进程/线程被唤醒, 但是最后只有一个进程/线程可以成功调用 accept 接收新连接, 其余的进程/线程会返回失败而重新休眠, 这样就产生了惊群现象.

针对 accept 惊群问题, Linux 2.6 版本给出了解决方案. 在 Linux 2.6 版本中, 维护了一个等待队列, 队列中的元素为进程, 并且使用了 WQ_FLAG_EXCLUSIVE 标志位 (互斥标志位). 当一个进程在调用 accept 函数时, 发生阻塞, 就会被标记为 exclusive (WQ_FLAG_EXCLUSIVE 标志位被置位). 非 exclusive 的进程会从前面添加到等待队列中, 而 exclusive 的进程会从后面添加到等待队列中. 当有新的连接到来时, 会遍历整个等待队列, 遍历到的每个进程 (这些都是非 exclusive 进程) 都会被唤醒, 直到遇到第一个 exclusive 进程. 这一方案保证了每次连接事件到来时只有一个进程被唤醒.

epoll 惊群

当不同的 epfd (epoll 实例的文件描述符, epoll 实例用于保存所有待监听的文件描述符) 监听同一个 socket 时, epoll 也可能出现惊群效应.

例如, 在 Nginx 的多进程架构中: 主进程 (Master 进程) 首先调用 listen 进行监听, 接着调用 fork 创建多个子进程 (Worker 进程), 子进程中调用 epoll_create 创建各自的 epfd, 并将继承自主进程的监听套接字添加到自己的 epfd 中, 再调用 epoll_wait 等待事件发生. 当连接请求到达时, 每个子进程的 epfd 都能捕捉到该事件, 这些子进程都会被唤醒, 但只有一个子进程可以通过 accept 争夺到该连接请求并建立连接. 如下图所示:

针对 epoll 惊群问题, 有三种解决方案:

  • accept_mutex.
  • EPOLLEXCLUSIVE.
  • SO_REUSEPORT.

accept_mutex

该方案由 Nginx 实现, Nginx 使用了一个全局锁, 当子进程在调用 epoll_ctl 和 epoll_wait 前, 需要获得该全局锁, 该策略确保了只有一个子进程在处理连接请求, 对应的指令是 accept_mutex.

EPOLLEXCLUSIVE

在 Linux 4.5 版本引入了 EPOLLEXCLUSIVE 标志位. 子进程调用 epoll_ctl 将监听套接字加入 epfd 时, 会同时将 EPOLLEXCLUSIVE 标志位显式传入, 这使得子进程带上了 exclusive 属性, 此时同一事件只会通知其中一个等待的线程或进程. 这一方案能够确保当一个事件发生时, 只唤醒一个正在 epoll_wait 阻塞的进程.

SO_REUSEPORT

在 Linux 3.9 版本引入了 socket 套接字选项 SO_REUSEPORT. 多个进程或线程可以使用 SO_REUSEPORT 选项创建和绑定 socket 到同一个端口, 内核会在这些 socket 之间进行负载均衡, 当连接请求到来时, 只有一个进程中的 socket 会获得连接.
