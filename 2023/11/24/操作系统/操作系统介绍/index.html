操作系统介绍

操作系统被设计出来, 旨在为多个用户应用程序, 提供比单一硬件所能支持的, 更多的、更有用的服务. 从整体的视图上看, 就是在用户软件和底层硬件间, 设计了一个中间层, 它负责管理底层的硬件设备, 抽象出各式各样的接口, 为用户程序提供一系列的标准服务, 而且不需要用户了解底层硬件的细节.

操作系统的目标

  • 将计算机的资源在多个程序间共享, 给程序提供服务.
  • 管理并抽象底层硬件.
  • 多路复用硬件, 使得多个程序可以同时运行, 但是彼此互不干扰.
  • 给程序提供一种受控的交互方式, 使得程序之间在需要时可以共享数据、共同工作.
  • 不阻止应用程序获得高性能, 甚至帮助应用程序获得高性能.
  • 能够支持大量不同类型的引用程序.

操作系统结构

  • 矩形表示计算机.
  • 硬件资源在最低层, 包含 CPU、内存、磁盘和网卡.
  • 在最高层, 运行各种应用程序, 这些正在运行的应用程序都运行在同一个空间中, 这个空间称为用户空间.
    • 每个用户空间程序都被称为进程, 它们有自己的内存和共享的 CPU 时间.
  • 内核 (Kernel) 是一个特殊的程序, 它总是在运行.
    • 当打开计算机时, 内核总是第一个被启动, 内核程序只有一个, 它维护数据来管理每个用户空间进程.
    • 内核还维护了大量的数据结构来帮助它管理各种硬件资源, 以供用户空间中的程序使用.
    • 内核拥有大量内置的服务
      • 文件系统, 管理文件内容并找出文件具体在磁盘中的位置, 用户空间的程序通过文件系统与磁盘的交互.
      • 进程管理系统, 管理内存的分配, 由于不同的进程需要不同大小的内存, 所以内核会复用内存、划分内存, 为所有的进程分配内存.
      • 访问管理系统, 当一个进程想要使用某些资源时, 访问管理系统会决定是否允许这样的操作.
      • …

操作系统接口

操作系统通过接口向用户程序提供服务. xv6 操作系统提供 Unix 操作系统中的基本接口, 同时模仿了 Unix 的内部设计.

xv6 使用了传统的内核形式 (一个向其他运行中程序提供服务的特殊程序).

每一个运行中程序 (称之为进程) 都拥有包含指令、数据和栈的内存空间. 进程通过系统调用使用内核服务, 系统调用会进入内核, 让内核执行服务然后返回, 所以进程总是在用户空间和内核空间之间交替运行.

内核使用 CPU 的硬件保护机制来保证每个在用户空间执行的进程只能访问它自己的内存.内核程序的执行拥有操控硬件的权限，它需要实现这些保护, 而用户程序执行时没有这些特权. 当用户程序调用系统调用时, 硬件会提升权限级别, 并开始执行内核中预先安排好的函数.

内核提供的系统调用集合是用户程序看到的接口, xv6 内核提供了 Unix 内核传统上提供的服务和系统调用的子集.

xv6 所有系统调用如下:

系统调用看起来和标准的函数调用相似, 但是两者存在很大的区别:

内核的代码总是具有特殊的权限, 当机器启动内核时, 内核会有特殊的权限能直接访问各种各样的硬件, 而普通的用户程序无法直接访问这些硬件, 所以当你执行一个普通的程序调用时, 你所调用的函数并没有对硬件的特殊权限.

进程和内存

xv6 进程由用户内存空间 (包含指令、数据和堆栈) 和对内核来说是私有的每个进程状态组成.

xv6 分时进程: 它透明地在等待执行的进程集合中切换可用的 CPU. 当一个进程没有执行时, xv6 保存它的寄存器文件, 并在下一次运行该进程时恢复它们. 内核利用进程 id 或 PID 标识每个进程.

I/O 和文件描述符

文件描述符是一个小整数, 表示进程可以读取或写入的由内核管理的对象. 进程可以通过打开一个文件、目录、设备, 或创建一个管道, 或复制一个已存在的描述符来获得一个文件描述符.

通常我们将文件描述符所指的对象称为 “文件”, 文件描述符接口将文件、管道和设备之间的差异抽象出来, 使得它们看起来就像字节流, 我们将输入和输出称为 I/O.

在内部, xv6 内核使用文件描述符作为每个进程表的索引, 每个表中保存文件描述符对应的实际内容. 每个进程都有一个从 0 开始的文件描述符的私有空间, 所以如果运行了两个不同的程序, 对应两个不同的进程, 如果它们都打开一个文件, 它们或许可以得到相同数字的文件描述符, 但是因为内核为每个进程都维护了一个独立的文件描述符空间, 这里相同数字的文件描述符可能会对应到不同的文件. 按照惯例, 进程从文件描述符 0 读取 (标准输入), 将输出写入文件描述符 1 (标准输出), 并将错误消息写入文件描述符 2 (标准错误).

Shell 利用上述惯例来实现 I/O 重定向和管道. Shell 确保它始终有三个打开的文件描述符, 这是控制台的默认文件描述符.

  • read 系统调用

    read 系统调用以字节为单位读取已打开的以文件描述符命名的文件.

    read(fd, buf, n) 从文件描述符 fd 读取最多 n 字节, 将它们复制到 buf, 并返回读取的字节数, 引用文件的每个文件描述符都有一个与之关联的偏移量. read 从当前文件偏移量开始读取数据, 然后将该偏移量前进所读取的字节数: 后续读取将返回第一次读取返回的字节之后的字节. 当没有更多的字节可读时, read 返回 0 来表示文件的结束.

  • write 系统调用

    write 系统调用以字节为单位写入已打开的以文件描述符命名的文件.

    write(fd，buf，n) 将 buf 中的 n 字节写入文件描述符, 并返回写入的字节数. 只有发生错误时才会写入小于 n 字节的数据. 与读一样, write 在当前文件偏移量处写入数据, 然后将该偏移量向前推进写入的字节数: 每个 write 从上一个偏移量停止的地方开始写入.

管道

管道是作为一对文件描述符公开给进程的小型内核缓冲区, 一个用于读取, 一个用于写入. 将数据写入管道的一端使得这些数据可以从管道的另一端读取, 管道是进程间通信的基本方式.

例如:

pipe 系统调用创建一个新的管道, 并在数组 p 中保存 read/write 文件描述符. 子进程调用 close 和 dup 使文件描述符 0 指向管道的 read 端, 然后关闭子进程的 p 中所保存的文件描述符, 并通过 exec 系统调用运行程序 wc. 当 wc 从它的标准输入读取 (文件描述符 0) 时, 就会从管道读取. 由于父进程不需要从管道读取数据, 所以关闭管道的read 端, 写入管道, 然后关闭管道的 write 端.

如果没有可用的数据, 则管道上的 read 操作将会进入等待, 直到有新数据写入或所有指向 write 端的文件描述符都被关闭, 在后一种情况下, read 将返回 0, 就像到达数据文件的末尾一样. 事实上, read 在新数据不可能到达前会一直阻塞, 这是子进程在执行上面的 wc 之前关闭管道的 write 端非常重要的一个原因: 如果 wc 的文件描述符之一指向管道的 write 端, wc 将永远看不到文件的结束.

管道相比临时文件的优势:

  • 管道会自动清理自己, 而文件重定向需要在使用完后删除临时文件.
  • 管道可以传递任意长的数据流, 而文件重定向需要磁盘上拥有足够的空闲空间来存储所有的数据.
  • 管道允许并行执行管道阶段, 而文件方法要求第一程序在第二个程序启动之前完成.
  • 如果实现进程间通讯, 管道的块读写比文件的非块语义更有效率.

当我输入 ls 时, 实际的意义是我要求 Shell 运行名为 ls 的程序, 文件系统中会有一个文件名为 ls, 这个文件中包含了一些计算机指令, 所以实际上, 当我输入 ls 时, 我是在要求 Shell 运行位于文件 ls 内的这些计算机指令.
