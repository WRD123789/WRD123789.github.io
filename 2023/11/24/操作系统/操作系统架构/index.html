操作系统架构

抽象系统资源

为了实现强隔离, 最好禁止应用程序直接访问敏感的硬件资源, 而是将资源抽象为服务.

  • Unix 应用程序只能通过文件系统的 open、read、write 和 close 系统调用与存储交互, 而不是直接读写磁盘. 文件系统对磁盘提供了非常方便的抽象. 你可以对文件命名, 读写文件等等. 之后, 操作系统会决定如何将文件与磁盘中的块对应, 确保一个磁盘块只出现在一个文件中, 并且确保用户 A 不能操作用户 B 的文件. 通过文件系统的抽象, 可以实现不同用户之间和同一个用户的不同进程之间的文件强隔离.
  • Unix 对 CPU 进行抽象, 在进程之间透明地切换硬件处理器, 根据需要保存和恢复寄存器状态, 这样操作系统才能在多个应用程序之间复用一个或者多个 CPU. 这种透明性允许操作系统共享处理器, 即使有些应用程序处于无限循环中.
  • Unix 使用 exec 对内存进行抽象, 进程使用 exec 来构建它们的内存映像, 而不是直接与物理内存交互. 这允许操作系统决定将一个进程放在内存中的哪里; 如果内存很紧张, 操作系统甚至可以将一个进程的一些数据存储在磁盘上. exec 还为用户提供了存储可执行程序映像的文件系统的便利.
  • Unix 进程之间的许多交互形式都是通过文件描述符实现的. 文件描述符不仅抽象了许多细节 (例如: 管道或文件中的数据存储在哪里), 而且还以简化交互的方式进行了定义(例如: 如果流水线中的一个应用程序失败了, 内核会为流水线中的下一个进程生成文件结束信号 (EOF)).

防御性

操作系统需要确保所有的组件能够工作, 所以它需要做好准备来抵御来自应用程序的攻击. 应用程序也不应该能够打破对它的隔离, 一旦攻击者打破对应用程序的隔离, 进而控制内核, 就会控制所有硬件资源.

所以需要在应用程序和操作系统之间提供强隔离性, 强隔离需要应用程序和操作系统之间的硬边界, 如果应用程序出错, 我们不希望操作系统失败或其他应用程序失败, 相反, 操作系统应该能够清理失败的应用程序, 并继续运行其他应用程序, 要实现强隔离, 操作系统必须保证应用程序不能修改 (甚至读取) 操作系统的数据结构和指令, 以及应用程序不能访问其他进程的内存, 这需要硬件来实现. CPU 为强隔离提供硬件支持, 硬件支持包括两个部分:

  • 用户/内核/机器模式 (user/kernel/machine mode).
    • 内核模式 (kernel mode/supervisor mode)
      • 在内核模式下, CPU 被允许执行特权指令, 特殊权限指令主要是那些直接操控硬件的指令和设置保护的指令.
      • 例如，启用和禁用中断、读取和写入保存页表地址的寄存器等.
      • 在处理器上有各种各样的状态, 操作系统会使用这些状态, 但是只能通过特殊权限指令来变更这些状态.
      • 处于内核模式下的软件可以执行特权指令, 在被称为内核空间的地方运行.
      • 在内核空间 (或内核模式) 中运行的软件被称为内核.
      • 处理器中有一个标志位, 当它为 1 时, 表示处于用户模式; 当它为 0 时, 表示处于内核模式. 当处理器解析指令时, 如果该指令是特殊权限指令且标志位为 1, 则处理器会拒绝执行该指令. 该标志位必须通过特殊权限指令进行更改.
    • 用户模式 (user mode)
      • 在用户模式下, CPU 只能执行普通权限的指令.
      • 例如, 将两个寄存器相加的指令 add、跳转指令 jal 等.
      • 应用程序只能执行用户模式的指令, 并在被称为用户空间的地方运行.
      • 如果用户模式下的应用程序试图执行特权指令, 那么 CPU不会执行该指令, 而是切换到内核模式, 以便内核模式代码可以终止应用程序.
    • 机器模式 (machine mode)
      • 在机器模式下执行的指令具有完全特权, CPU 在机器模式下启动.
      • 机器模式主要用于配置计算机.
      • xv6 在机器模式下执行很少的几行代码, 然后更改为管理模式.
    • 想要调用内核函数的应用程序 必须过渡到内核. CPU 提供一个特殊的指令, 将 CPU 从用户模式切换到内核模式, 并在内核指定的入口点进入内核 (RISC-V 为此提供 ecall 指令). 一旦CPU 切换到内核模式, 内核就可以验证系统调用的参数, 决定是否允许应用程序执行请求的操作, 然后拒绝它或执行它. 由内核控制转换到内核模式的入口点是很重要的, 如果应用程序可以决定内核入口点, 那么恶意应用程序就可以在跳过参数验证的地方进入内核.
  • 虚拟内存 (Virtual Memory).
    • 处理器包含页表, 页表将虚拟内存地址映射到物理内存地址.
    • 每个进程都有自己独立的页表, 这样每个进程只能访问出现在自己页表中的物理内存. 操作系统会设置页表, 使得每个进程都有不重合的物理内存, 这样一个进程就不能访问其他进程的物理内存.
    • 页表定义了对于内存的视图, 而每个用户进程都有自己对于内存的独立视图.

所有的处理器, 如果需要运行能够支持多个应用程序的操作系统, 就需要同时支持用户/内核/机器模式和虚拟内存.

模式切换

我们可以认为用户/内核模式是分隔用户空间和内核空间的边界, 用户空间运行的程序运行在用户模式, 内核空间的程序运行在内核模式, 操作系统位于内核空间.

在 RISC-V 中, 通过 ecall 指令让应用程序将控制权转移给内核. ecall 接收一个数字参数, 当一个用户程序想要将程序执行的控制权交给内核时, 它只需要执行 ecall 指令, 并传入一个数字, 这个数字决定应用程序想要调用的系统调用.

内核组织

内核负责实现具体的系统调用并检查参数以确保不会被一些坏的参数所欺骗, 所以内核有时候也被称为可被信任的计算空间 (Trusted Computing Base, TCB).

根据操作系统的哪些部分应该以内核模式运行可以将操作系统的设计分为两类

  • 宏内核
    • 整个操作系统都驻留在内核中, 即所有系统调用的实现都以内核模式运行.
    • 在这个组织中, 整个操作系统以完全的硬件特权运行.
    • 这个组织比较方便, 操作系统的设计者不需要考虑操作系统的哪一部分不需要完全的硬件特权.
    • 因为操作系统的各个部分都在内核空间中, 他们可以更紧密地集成在一起, 所以操作系统的不同部分更容易合作, 拥有更好的性能.
    • 操作系统的不同部分之间的接口很复杂, 对于操作系统的设计者而言, 更容易犯错误. 因为任何操作系统中的错误都会导致内核错误, 从而影响整个计算机的工作, 随着在内核中的操作系统的规模的增加, 出现错误的概率也会增大.
  • 微内核
    • 为了降低内核出错的风险, 操作系统设计者可以最大限度地减少在内核模式下运行的操作系统的代码量, 并在用户模式下执行大部分操作系统.
    • 微内核设计中, 可以将文件系统作为用户级进程运行. 作为用户级进程运行的操作系统服务被称为服务器. 为了允许应用程序与文件服务器交互, 内核提供了允许从一个用户态进程向另一个用户态进程发送消息的进程间通信机制 (IPC 系统).
      • 例如, 如果像 Shell 这样的应用程序想要读取或写入文件, 它会向文件服务器发送消息并等待响应.
    • 由于两个用户进程通过内核上的 IPC 系统进行交互, 所以应用程序与部分操作系统服务的交互需要完成两次用户空间 ⇐⇒\Leftarrow \Rightarrow⇐⇒ 内核空间的跳转; 而对于宏内核, 应用程序与部分操作系统服务的交互只需要用户空间 ⇐⇒\Leftarrow \Rightarrow⇐⇒ 内核空间的跳转, 所以微内核的性能更差.
    • 在一个类似宏内核的紧耦合系统中, 各个组合部分可以很容易的共享页缓存; 而在微内核中, 各个部分之间很好地分隔开了, 这种共享更难实现, 所以微内核难以得到高性能.
    • 由于客户/服务器 (Client/Server) 模式, 具有非常多的优点, 故在单机微内核操作系统中几乎无一例外地都采用客户/服务器模式, 将操作系统中最基本的部分放入内核中, 而把操作系统的绝大部分功能都放在微内核外面的一组服务器(进程)中.

进程

操作系统中的隔离单位是一个进程, 进程抽象防止一个进程破坏或监视另一个进程的内存、CPU、文件描述符等. 它还防止一个进程破坏内核本身，这样一个进程就不能破坏内核的隔离机制.

内核用来实现进程的机制包括用户/管理模式标志、地址空间和线程的时间切片.

为了帮助加强隔离, 进程抽象给程序一种错觉, 即让它认为自己独占 CPU 且拥有自己的地址空间.

xv6 使用页表 (由硬件实现) 为每个进程提供自己的地址空间. RISC-V 页表将虚拟地址 (RISC-V 指令操控的地址) 映射到物理地址 (内存地址).

xv6 为每个进程维护一个单独的页表, 该页表定义了该进程的地址空间, 该地址空间以虚拟内存地址 0 开始, 首先是指令, 然后是全局变量, 然后是用户栈, 然后是堆, 在地址空间的顶部, xv6 为 trampoline (用于在用户和内核之间切换) 和映射进程切换到内核的 trapframe 分别保留了一个页面.

有许多因素限制了进程地址空间的最大范围: RISC-V 的指针由 64 位宽, 硬件在页表中查找虚拟地址时只使用低 39 位, 而 xv6 只使用这 39 位中的 38 位, 所以最大虚拟内存地址 (MAXVA) 为 238−1=0x3fffffffff2^{38} - 1 = 0x3fffffffff238−1=0x3fffffffff.

xv6 内核为每个进程维护许多状态片段, 并将它们聚集到一个 proc 结构体中. 一个进程最重要的内核状态片段是它的页表、内核栈区和运行状态.

每个进程都有一个执行线程 (简称线程) 来执行进程的指令. 一个线程可以挂起并且稍后再恢复. 为了透明地在进程之间切换, 内核挂起当前运行的线程, 并恢复另一个进程的线程.

线程的大部分状态（本地变量、函数调用返回地址）存储在线程的栈区上. 每个进程有两个栈区: 一个用户栈区和一个内核栈区.

当进程执行用户指令时, 只有它的用户栈在使用, 它的内核栈是空的; 当进程进入内核 (由于系统调用或中断) 时, 内核代码在进程的内核堆栈上执行. 当一个进程在内核中时, 它的用户堆栈仍然包含保存的数据, 只是不处于活动状态.

进程的线程在主动使用它的用户栈和内核栈之间交替. 内核栈是独立的 (受用户代码的保护), 因此即使一个进程破坏了它的用户栈, 内核依然可以正常运行.

一个进程可以通过执行 RISC-V 的 ecall 指令进行系统调用, 该指令提升硬件特权级别, 并将程序计数器 (PC) 更改为内核定义的入口点, 入口点的代码切换内核栈, 执行实现系统调用的内核指令, 当系统调用完成时, 内核切换回用户栈, 并通过调用 sret 指令返回用户空间, 该指令降低了硬件特权级别, 并在系统调用指令刚结束时恢复执行用户指令.

进程的线程可以在内核中 “阻塞” 等待 I/O, 并在 I/O 完成后恢复到中断的位置.

进程状态有: 已分配、就绪态、运行态、阻塞态 (等待 I/O 中) 和退出.

内核编译

xv6 代码主要由三部分组成

  • kernel
    • 里面包含基本所有的内核文件. 因为 xv6 是一个宏内核结构, 所以所有的文件都会被编译成一个 kernel 的二进制文件, 然后这个二进制文件会被运行在内核模式中.
  • user
    • 包含运行在用户模式的所有程序.
  • mkfs
    • 它会创建一个空的文件镜像, 我们会将这个镜像存储在磁盘上, 这样我们就可以直接使用一个空的文件系统.

编译过程

  • Makefile 会读取所有的 C 文件, 通过 gcc 编译器, 生成以 .s 为后缀的 RISC-V 汇编语言文件, 之后通过汇编解释器生成以 .o 为后缀的二进制目标文件, 然后系统加载器会将所有的 .o 文件链接起来, 生成内核文件.

  • Makefile 还会创建 kernel.asm 文件, 其中包含内核的完整汇编语言代码.

  • 在 QEMU (一个用 C 语言来模拟仿真 RISC-V 处理器的程序) 上运行 xv6.

    • -kernel: 传递的是内核文件 (kernel 目录下的 kernel 文件), 这是将要在 QEMU 中运行的程序文件.
    • -m: 传递的是 RISC-V 虚拟机将要使用的内存数.
    • -smp: 传递的是虚拟机可以使用的 CPU 核数.
    • -drive: 传递的是虚拟机使用的磁盘驱动, 这里传入的是fs.img 文件.

QEMU

QEMU 是一个 C 程序, 但它表现的就像一个真正的 RISC-V 主板一样.

在 RISC-V 主板上可以启动一个 xv6, xv6 的结构图如下:

  • 4 个核: U54 Core 1-4
  • L2 Cache: Banked L2
  • 连接 DRAM 的连接器: DDR Controller
  • 各种连接外部设备的方式
  • 连接时钟的接口: Clock Generation

通过 QEMU 模拟的计算机主板, 和上述 RISC-V 主板非常相似. 对于 CPU 中的每个核, QEMU 都会运行这样一个循环:

  • 读取 4 字节或 8 字节的 RISC-V 指令.
  • 解析 RISC-V 指令.
  • 在软件中执行相应的指令.

为了完成工作, QEMU 需要维护寄存器的状态, 所以会有以 C 语言声明的类似于 x0, x1 的寄存器.

当 QEMU 在仿真 4 个 CPU 核时, 它是并行的模拟这 4 个核.

内核启动过程

当 RISC-V 计算机上电时, 它会初始化自己并运行一个存储在只读内存 (启动扇区) 中的引导加载程序 (bootblock, 因为启动扇区的空间比较小, 不足以存储内核代码, 所以需要一个引导加载程序来帮助加载内核代码). 引导加载程序将 xv6 内核加载到内存中, 然后在机器模式下, 中央处理器从 _entry 开始运行 xv6, xv6 启动时页式硬件 (Paging Hardware) 处于禁用模式: 虚拟地址将直接映射到物理地址.

  • 其中引导加载程序会将 xv6 内核加载到物理地址为 0x80000000 的内存中, 而不是 0x0 的位置, 因为地址范围 0x0 : 0x80000000 包含 I/O 设备.

_entry 的指令设置了一个栈区, 这样 xv6 就可以运行 C 代码. xv6 在 kernel/start.c 文件中为初始栈 stack0 声明空间. 由于 RISC-V 上的栈是向下扩展的, 所以_entry 的代码将栈顶地址 stack0 + 4096 加载到栈顶指针寄存器 sp 中. 当内核拥有栈区后, _entry 便调用 start 函数 (kernel/start.c).

函数 start 执行一些仅在机器模式下允许的配置，然后切换到内核模式. start 函数执行以下操作:

  • 在机器态特权寄存器 mstatus 中将先前的运行模式切换为内核模式.
  • 将 main 函数的地址写入寄存器 mepc 中, 即将返回地址设为 main.
  • 将 0 写入页表寄存器 satp 中, 即在内核模式下禁用虚拟地址转换.
  • 对时钟芯片进行编程以产生计时器中断.
  • 将所有的中断和异常委托给内核模式.
  • 通过调用 mret 返回到内核模式, 这将导致程序计数器的值更改为 main (kernel/main.c) 函数的地址.
    • mret 最常用于从内核模式切换到机器模式的调用中返回, 由于 start 函数将先前运行模式设置为内核模式, 且将返回地址设置为 main, 因此 mret 将返回到 main 函数, 并以内核模式运行.

在 main 函数 (kernel/main.c) 初始化几个设备和子系统后, 便通过调用 userinit (kernel/proc.c) 创建第一个进程 (进行该进程的初始化工作, 但并不运行), 创建第一个进程后, 进入 scheduler 函数, 该函数为进程调度器, 调度第一个进程, 第一个进程执行一个用 RISC-V 程序集写的小型程序 initcode.S (uer/initcode.S), 它通过调用 exec 系统调用重新进入内核. 此时 exec 用 /init 程序替代当前进程的内存和寄存器, 所以 initcode.S 程序只是用来启动 init 程序. init (user/init.c) 将创建一个新的控制台设备文件, 然后以文件描述符 0、1 和 2 打开它, 然后它在控制台上通过 fork + exec 来启动一个 Shell 程序, 这样系统的启动了.

其中 init 是第一个进程 (0 号进程), 它被存放到磁盘上, 它是所有进程的祖先; Shell 是第二个进程 (1 号进程).

程序 位置 用途
initcode 内存代码段 调用 exec 启动 init 程序
init 磁盘 启动 Shell, 回收僵尸进程
Shell 磁盘 用户操作
