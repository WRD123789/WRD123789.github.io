锁

由于我们有多个 CPU, 这些 CPU 各自独立地执行指令, 并且共用 RAM, 因此我们可以在多个 CPU 之间维护一些共享数据结构, 但这会导致一些问题.

  • 例如: 在一个 CPU 读取数据时, 另一个 CPU 可能正在更新数据, 如果我们不加以干涉, 则会导致读取的数据错误.

访问共享资源的代码段, 我们称之为临界区, 而多个执行线程同时进入临界区并尝试更新共享资源的情况, 我们称之为竞争条件.

即使是单 CPU, 也会在多个进程之间切换执行, 使这些进程的执行序列变得交错.

  • 例如: 当中断开放时, 如果某中断处理程序正在修改某些数据, 而进行到一半时又被新的中断介入, 该数据就会出现问题.

因此, 共享数据结构在多 CPU 或单 CPU 的情况下, 都可能会出现问题.

以上是我们需要考虑的并发问题, 它们通常由多个执行指令流交错而产生, 可进一步归结为以下原因:

  • 多处理器的并行执行
  • 线程切换
  • 中断

内核之中总是遍布着各种共享数据结构, 我们希望有一种兼具并发性和正确性的抽象设计, 线程应该使用这种设计提供的互斥原语, 使得多个执行线程可以同时进入临界区, 避免产生竞争条件, 最终能够产生确定的输出. 为此设计者提供了很多并发控制技术 (也称为同步工具).

常用的同步工具有: 锁、条件变量、信号量等.

这些同步工具保证了临界区中工作的原子性.

其中锁是一种互斥原语, 一次只有一个 CPU 能够获得锁. 因此, 如果我们给一个临界区上锁, 那么一次只能有一个 CPU 在临界区中执行, 锁成功地保护了临界区中的共享数据结构. 尽管使用锁可以解决很多的并发问题, 但如果使用不当的话, 会影响性能表现, 因为锁本质上使并发操作变得串行化 (也称为序列化), 当然也会有多个 CPU 争夺同一把锁的情况.

总的来说, 锁的作用为:

  • 有助于避免更新的丢失.
  • 能够使多步操作变为原子性的操作.
  • 有助于维持一些规则或特性的不变性.

  • 互斥: 一次只能有一个进程进入临界区中, 并在其中执行.
  • 进步: 如果当前没有进程在临界区中执行, 且有多个进程同时想要进入临界区, 应该有某种方式来决定谁先进入, 而且这种决定不能被无限推迟.
  • 有限等待/无饥饿: 进程从请求进入临界区开始, 直到被允许进入临界区的这段时间应该是有限的, 即每个竞争进程都应该有公平的机会抢到锁, 不能有竞争锁的进程处于饥饿 (一直无法获得锁).

有两种用于处理操作系统临界问题的常用方法:

  • 抢占式内核: 允许在内核空间下运行的进程的 CPU 被抢占.
  • 非抢占式内核: 不允许进程的 CPU 被抢占, 进程会一直运行知道退出内核空间、阻塞或自愿放弃 CPU.

非抢占式内核基本不会导致竞争条件, 因为任一时刻只有一个进程在内核空间下运行; 而抢占式内核则需要仔细设计, 以防止竞争条件出现, 尤其是对称多处理器体系结构. 但我们更倾向于使用抢占式内核, 因为这种内核响应更快, 进程不会在内核中运行任意长的时间, 还允许实时进程抢占在内核空间下运行的其它进程.

Peterson 算法

Peterson 算法是一个基于软件的临界区问题的解决方案.

该算法的主要思想是谦让, 一个线程总是试图把进入临界区的权利让给另一个线程. 假设当前线程首先更新 flag, 表示自己想要进入临界区, 但是我们会先把机会让给另一个线程, 希望对方优先进入临界区. 如果对方刚好也更新 flag, 则对方就能顺利进入临界区, 而当前进程就在原地等待; 如果对方没有更新 flag, 表示它不想进入临界区, 或者对方先谦让给我们, 那就轮到我们顺利地进入临界区.

内存屏障

计算机体系结构将确定它为应用程序提供什么样的内存保证, 这被称为内存模型. 一般来说, 内存模型可以分为两类:

  • 严格一致性/强一致性: 一个 CPU 对内存做了某些修改, 其他 CPU 马上就能看到这些修改.
  • 松散一致性/弱一致性: 一个 CPU 对内存做了某些修改, 但其他 CPU 并不能马上看到这些修改, 可能在稍后才能看到这些修改.

由于内存模型的类型和 CPU 密切相关, 在一个共享 RAM 的多处理器系统中, 我们的内核不能对这些修改的可见性做任何假设. 为此, 计算机体系结构提供了一种指令, 它可以强制内存的更新对所有 CPU 可见, 这种指令就是内存屏障 (Memory Barriers).

当执行内存屏障指令时, 系统确保, 在内存屏障之前所有的 load 和 store , 都会在内存屏障之后的 load 或 store 执行前完成. 因此, 即使指令被重新排序, 内存屏障确保了, 位于它之前的存储操作已经在内存中完成, 因此在内存屏障之后, 这些存储操作对所有处理器是可见的.

自旋锁不满足有限等待的要求, 可能会导致某些线程饥饿, 且自旋锁主要的缺点是会忙等, 这种忙等会持续消耗 CPU 资源, 因为它要不断地旋转以等待锁可用.

  • 对于单 CPU 来说, 自旋锁的性能开销相当大, 假设一个线程持有锁并进入临界区, 然后因为时钟中断被抢占, 调度器因此运行其他线程, 而其他线程也可能竞争该锁, 由于被中断的线程还未释放锁, 所以这些竞争锁的线程在时钟中断到达前 (时间片到期前) 会不停地自旋等待锁可用, 从而浪费了大量 CPU 的周期.
  • 对于多 CPU 来说, 尤其是线程数大于 CPU 数时, 自旋锁的性能表现不错. 假设线程 A 在 CPU1 上, 线程 B 在 CPU2 上, 都在竞争同一个锁, A 占有锁时, B 会在 CPU2 上自旋, 由于临界区一般很短, 所以 B 很快就能获得锁. 因此自旋等待锁的线程不会浪费太多 CPU 周期.

线程在等待锁时没有发生上下文切换 (上下文切换的开销可能很大), 因此如果临界区较短, 使用自旋锁时一个不错的方案.

锁和中断

最早的互斥解决方案是为单 CPU 系统开发设计的: 在进入临界区之前关闭中断, 可以保证 临界区内的代码原子地执行, 离开临界区后在重新开启中断.

  • 这个方法的优点是简答, 但是缺点很多:
    • 开关中断的指令是特权指令, 这要求我们给予用户线程高特权级别, 这会带来很多问题.
    • 不适用于多处理器, 当多个线程运行在不同的 CPU 上时, 即使关闭中断也不能阻止多个线程访问同一个临界区.
    • 关闭中断可能会导致中断丢失.
      • 例如: 磁盘设备完成了读取请求发出中断, 但由于关闭中断而错过了该中断.
    • 开关中断指令执行较慢.

对于自旋锁, 当线程和中断处理程序要获取同一把自旋锁时, 会出现问题.

  • 例如: 在 Xv6 中, 时钟中断处理程序 clockintr 需要增加 ticks 值, 而一个内核线程也可能通过系统调用 sys_sleep 来访问 ticks 值, 因此, 我们为 ticks 维护了一把锁 tickslock, clockintr 和 sys_sleep 都需要获取该自旋锁来访问或修改 ticks 值. 假设一个内核线程调用 sys_sleep, 此时它持有 tickslock, 此时运行该内核线程的 CPU 收到时钟中断, 因此执行中断处理程序 clockintr, clockintr 首先需要获取 tickslock, 但是这把锁已经被内核线程持有, 所以 clockintr 会一直在原地自旋等待 tickslock 被释放. 而 tickslock 无法被内核线程释放, 该内核线程无法调用 yield, 该内核线程的状态仍然为 RUNNING, 所以该内核线程也无法被调度, 此时该 CPU 发生死锁.

因此为了避免这种情况的发生, 我们应该做以下规定:

  • 如果中断处理程序需要持有某一把自旋锁, 那么每个 CPU 在持有这把锁时应该保持中断关闭.

睡眠锁

我们希望如果一个锁在长时间内都被其他进程持有时, 尝试获取该锁的其他进程不必占用大量的 CPU 周期, 而是以一种睡眠的方式主动放弃 CPU, 在稍后的某个时间再被重新唤醒执行. 这种锁会在 acquire 需要自旋等待时主动让出 CPU, 同时在持有这种锁时, 允许主动放弃 CPU 和开放中断.

一个简单的方案如下:

  • 我们假设内核提供 yield 原语, 进程可以调用它主动放弃 CPU.
  • 进程的运行状态可以按以下的 enum 体来定义:
    • yield 让进程的状态从 RUNNING 变为 RUNNABLE, 并让调度器寻找其他状态为 RUNNABLE 的进程.
    • 该方案不能明确地支持有限等待条件, 所以我们需要显示地增加某种机制: 使用队列来保存等待锁的进程
      • 这里实际上使用两把锁, 一个自旋锁 guard lock (小锁), 另一个锁是真正的临界区锁 flag lock (大锁). 这个方法虽然没有完全避免自旋等待, 但是这个自旋等待的时间很有限, 因为 guard lock 不是保护临界区的, 只是保护 lock 和 unlock 之间的几行代码.
      • 但是这个方案还存在潜在的竞争条件: 假设 thread1 已经持有了 flag lock, 后续 thread2 尝试获取 flag lock 时就会陷入到 lock() 的 else 分支中, 将自己添加到队列之后, 但在调用 park() 之前被切换到 thread1 , thread1 执行 unlock 时, 由于队列不为空, 将会执行 unpark(thread2), 但 thread2 的 park() 并未执行, 此时可能返回 thread2 并未睡眠的消息, 然后 thread1 退出临界区, thread2 继续执行并 park() 自身, 进入睡眠. 但是 thread1 并没有将 flag lock 重置为 0, 因此后续尝试获取该锁的线程都将进入睡眠, 这种情况称为唤醒/等待竞争.
        • Solaris 增加了第三个系统调用 setpark 来解决这种情况, 一个线程通过调用 setpark 表明自己马上要 park, 此时如果刚好另一个线程被调度, 并且调用了 unpark, 那么后续的 park 调用就会直接返回.

  • Xv6 中同样采用了自旋锁和睡眠所相结合的形式.
  • 由于 Xv6 的睡眠锁的实现允许中断开放, 所以中断处理程序不能使用睡眠锁, 否则会导致死锁.

睡眠锁能够避免使用自旋锁时带来的大量 CPU 周期的浪费, 但是来回地切换进程会导致较高的上下文切换开销.

所以在临界区较短时, 使用自旋锁比较合适; 而在临界区较长时, 使用睡眠锁更合适. 也可以将两种所结合起来, 称为二阶段锁, 第一阶段自旋一段时间 (自旋固定的次数), 希望可以获取锁, 但是如果没有获得锁, 则在第二阶段会睡眠, 直到锁可用.

锁的争夺

利用互斥锁, 我们确实解决了大部分的并发问题, 现在我们的临界区总是能正确运行; 但是使用锁确实会使并发问题变得串行化, 一次只能有一个进程在临界区中运行, 从而影响性能. 我们因为使用了多 CPU 和多线程而产生并发问题, 而为了解决并发问题, 我们使用了锁, 这又使得多CPU 的性能优势衰减, 经常出现多个进程争夺锁的情况, 我们称这种现象为锁的争夺.

内核设计中, 一个重要的部分就是如何避免锁的争夺.

  • 在 Xv6 中, 内核分配器的 free-list (保存空闲物理页的一个链表) 只用一把锁保护, 因此多个 CPU 想要分配和释放物理页需要争夺同一把锁. 我们可以为每个 CPU 都维护一个 free-list, 因此锁的数量从原来的一把增加到和 CPU 相等的数量, 这样 CPU 在大部分情况下不需要和其他 CPU 争夺锁, 只有在自己的 free-list 用完后才需要和其他 CPU 争夺锁. 这种方式可以大量减少锁的争夺. 同样, 对于只有一把锁的缓冲区, 我们可以用类似的思路, 将缓冲区的空间分配到多个哈希桶中, 每个哈希桶分配一把锁, 根据块号索引哈希桶, 从而减少缓冲区上锁的争夺.

这表明在实际设计并发代码时, 我们可以先采用粗粒度的上锁方式 (只有一个线程可以同时访问受锁保护的共享资源, 例如只给内核分配器的 free-list 上一把锁), 先给临界区上一把大锁. 然后在保证并发正确的情况下采用细粒度的上锁方式 (即将共享资源分割成多个小的部分, 并给每个部分分配一个独立的锁, 以允许多个线程同时访问不同的资源部分, 例如给每个 CPU 都维护一个 free-list, 然后对每个 CPU 的 free-list 单独上锁). 采用何种粒度的上锁方式, 取决于你对性能和代码的复杂性的要求.

死锁

一般来说, 死锁的产生需要满足以下四个条件, 只要它们同时成立, 就会引起死锁:

  • 互斥: 至少有一个资源处于非共享模式, 即进程对于需要的资源进行互斥的访问.
  • 占有并且等待: 进程至少占有一个资源, 并等待其他资源, 而该资源被其他进程所占有.
  • 非抢占: 进程获得的资源不能被抢占.
  • 循环等待: 进程之间存在一个环路, 环路上每个进程都额外持有一个资源, 而这个资源是下一个进程要申请的.

以上四个条件由任意一个不满足, 就不发生死锁.

预防死锁

预防死锁最好的方法就是破坏四个条件中的任意一个:

  • 循环等待: 最直接的方法时提供一个全局的上锁顺序, 即全序. 更复杂的系统可能有很多个锁, 锁的全局就很难做到, 因此偏序也是一种有效的方法. 全序和偏序都依赖于细致的锁策略设计和实现.
  • 占有且等待: 可以让每个进程在执行之前先申请并成功获得所需的所有资源. 此时我们可以先设置一把大锁, 进程要申请资源时先获取大锁, 成功持有大锁后, 再去获取所需资源的锁, 然后再释放大锁. 这个方法有不少缺点:
    • 不适合封装,因为我们需要确切地知道要获取哪些锁, 并且提前获取这些锁.
    • 因为要提前获取所有的锁, 而不是在需要的时候获取锁, 这会降低并发性.
    • 会降低资源利用率, 因为需要资源被分配, 但是可能在很长一段时间内不被使用.
    • 可能会导致饥饿, 如果一个进程需要多个常用资源, 那么它可能会长久地等待, 因为它所需要的资源总是至少有一个被分配给其他进程.
  • 非抢占: 可以设计这样一种协议: 如果一个进程持有资源并申请另一个不能立即分配的资源, 并且即将进入等待状态, 那么它现在持有的资源可以被抢占. 在实现这种协议时, 我们可以引入一个新的接口 trylock.
    • trylock 会尝试获取某把互斥锁, 如果发现它被占有, 那么就返回, 稍后再尝试获取它.
    • 利用 trylock, 我们可以先获取一把互斥锁, 在获取第二把的时候使用 trylock, 如果 trylock 返回的结果是失败, 我们就释放持有的第一把互斥锁.
    • 但是这种设计存在缺陷: 假设有两把互斥锁 L1 和 L2, 一个线程 lock(L1) →\rightarrow→ trylock(L2), 另一个线程 lock(L2) →\rightarrow→ trylock (L1). 如果两个线程的第一步同时成功, 那么它们将分别持有 L1 和 L2, 这时候两个线程都尝试 trylock, 此时两者都会失败, 于是各自释放 L1 和 L2. 在下一次尝试中它们有可能又遇到相同的情况, 如此反复. 这种现象称为活锁. 解决方法是在 trylock 失败并且释放持有的第一把锁之后, 随机等待一段时间再重复整个动作, 这样就可以降低线程之间互相干扰的可能.
  • 互斥: 可以使用无等待的数据结构, 可以利用硬件指令来构造.

避免死锁

死锁预防的四种方法, 主要通过限制如何申请资源来预防死锁, 但可能造成设备使用率低和系统吞吐率低. 死锁避免也是一种方法, 这种方法需要全局信息, 包括不同进程或线程在运行中对锁或资源的需求情况, 从而合理地调度并分配资源, 使得死锁不会发生.

常用的死锁避免算法有: 安全状态算法、资源分配图算法、银行家算法等; 控制进程或线程的调度也是一种可能的方法.

死锁的检测和恢复

死锁检测一般通过构建资源分配图或等待图来进行, 检测图中是否存在循环, 就能判断死锁是否发生.

死锁恢复有几种方法. 例如: 简单地终止一个或多个进程来打破循环等待; 或者, 从一个或多个死锁进程处, 抢占一个或多个资源; 重启也不失为一种方法.

很多数据库使用了死锁检测和恢复技术. 死锁检测器会定期运行, 构建资源分配图或等待图, 检查图内是否存在循环, 从而判断是否发生死锁. 若死锁发生, 系统可能需要重启, 如果更复杂的数据结构需要修复, 则需要人工介入.

死锁与上锁顺序

死锁的出现可能与上锁顺序有关

  • 例如: 假设进程 1 获取锁的顺序是先 A 后 B, 进程 2 获取锁的顺序是先 B 后 A. 如果进程 1 获取 A 的同时进程 2 获取了 B, 则此时会发生死锁.

想要避免这种死锁, 只需要规定一个全局的上锁顺序, 这意味着锁是每个函数规范的一部分: 调用者必须以一种规定的顺序调用函数, 以遵循获取锁的顺序.

但是遵循一个全局的上锁顺序是十分困难的, 首先确定全局上锁顺序需要对程序进行仔细的分析和设计, 需要考虑多种情况和可能性, 比较繁琐和复杂. 其次, 全局上锁顺序可能会降低程序的并发性能, 因为所有的线程或进程都必须按照同样的顺序获取锁, 会造成一定的等待和延迟.

  • 例如, 如果一个模块 m1 中方法 g 调用了另一个模块 m2 中的方法 f, 那么 m1 中的方法 g 需要知道 m2 中的方法 f 使用了哪些锁. 因为如果 m2 使用了一些锁, 那么 m1 的方法 g 必须要整合 f 和 g 中的锁以形成一个全局的上锁顺序, 这意味着 m2 中的锁必须对 m1 可见, 这样却违背了代码抽象的原则, 所以这使得代码的模块化更加复杂.

如果两组锁不可能在同一个操作中使用, 那么这两组锁的排序是完全独立的. 所以没有必要对所有的锁进行一个全局的排序, 但是所有的函数需要对共同使用的一些锁进行排序.
