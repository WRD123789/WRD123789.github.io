哈希码

哈希码将一个包含很多甚至无限元素的集合中单个值投影到一个有固定数目的集合中的值.

用类似十进制的方式

将每个字符替换为其 ASCII 码:

然后计算每个字符的 ASCII 码 * 其底数 (此处为 ASCII 码的总数) 的 i 次幂, i 为该字符在字符串中从左数的位置, i 从 0 开始.

例如:

字符串 bee 的哈希码 = (98×1272)+(101×1271)+(101×1270)(98 \times 127^2) + (101 \times 127^1) + (101 \times 127^0)(98×1272)+(101×1271)+(101×1270) = 159357015935701593570.

该方法存在一定问题, 当产生的哈希码很大, 大到一定程度时, 会超出 int 的上限 ( Java 中最大的 int 为 2147483647 ) , 此时发生溢出.

溢出意味着从最小的数开始 ( Java 中 2147483647 + 1 后变为 -2147483648 ), 那么这个时候就可能发生碰撞, 即多个字符串产生了相同的哈希码.

解决碰撞

当发生碰撞时, 即多个项目对应同一个哈希码时, 可以让每个数组存储链表.

此时有两个选择:

  • 牺牲空间追求时间:

    数组对于每个哈希码, 都存储一个链表:

    这样每次查找和插入的时间不再是 θ(1)\theta (1)θ(1), 而是 O(Q) , 其中 Q 为数组中最长的链表的长度. 最坏情况下所有项目放在同一个链表中, 此时的时间复杂度为 θ(N)\theta (N)θ(N); 最好情况每个链表只存储一个项目, 此时的时间复杂度为 θ(1)\theta (1)θ(1), 所以总的时间复杂度为 O(N) .

  • 牺牲时间追求空间

    每个项目存储在 哈希码 % 10 的位置, 则数组只需要开 10 个空间, 总共只有 10 个链表, 但是每个链表的长度会大幅度增加, 由于每次查找和插入的时间复杂度为 O(Q) , 即操作的时间与链表的长度有关, 所以操作总的时间相比第一种操作也会大幅度增加. 最坏情况下所有项目放再同一个链表中, 所以此时的时间复杂度仍为 θ(N)\theta (N)θ(N); 最好情况下所有项目均匀地分配到每个链表中, 此时链表的长度为 N / 10 , 所以此时的时间复杂度为 θ(N)\theta (N)θ(N), 所以总的时间复杂度为 θ(N)\theta(N)θ(N).
