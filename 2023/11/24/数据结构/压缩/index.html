压缩

Shannon-Fano 编码

  • 统计文本中所有字符出现的概率.
  • 每次分成概率大致相等的两部分
    • 左边得到前导 0, 右边得到前导 1.
    • 重复上述步骤.

Huffman 编码

过程

  • 计算文本中所有字符出现的概率.
  • 将每个字符放入权重为该字符出现概率的节点.
  • 每次将两个权重 (概率) 最小的节点合并成一个节点, 该节点的权重等于两个被合并的节点的权重之和.
  • 重复上一步骤直到所有节点在同一棵树中.

而 Shannon-Fano 编码对上述输入数据编码后每个字符的平均位数 = 2.31, 可见 Huffman 编码优于 Shannon-Fano 编码.

数据结构选择

编码选择用数组存储来存储比特流, 用字符作为索引 (例如: ‘A’ 为 65), 因为如果用哈希表来存储, 需要计算哈希码, 需要运用模运算 (模运算比较花时间), 不如用数组存储更快:

解码选择用字典树:

还需要考虑 Huffman 编码如何存储, 主要有两种方式:

  • 为每种输入类型汇编大量示例输入, 根据文献为英文、汉语等创建标准编码.
  • 为每个待压缩文件单独创建编码, 并将该码字表与压缩文件存储在一起.

第一种方法创建的编码不一定是最优的, 但由于已经构建好了码字表, 所以运行更快; 第二种方法需要为压缩比特流产生的码字表准备额外的空间, 同时创建码字表需要更多的工作, 所以速度相对第一种方法会更慢.

对于大的输入, 包含码字表的成本便微不足道, 所以日常生活中采取第二种方式.

压缩理论

Huffman 编码的主要思想是用少量的比特表示字符.

压缩的总体思想是利用序列中的冗余和现有顺序.

不同的压缩算法在不同的文件上实现不同的压缩比.

为什么没有压缩算法可以将任何比特流压缩 50%: 假设如果有压缩算法能够将任何比特流压缩 50%, 那么就可以对该比特流一直调用该压缩算法, 最终比特流会被压缩到只有一位, 这显然不可能.

即使只能压缩一次, 也不可能将任何比特流压缩 50%: 因为短比特流远远小于长比特流. 假设有 210002^100021000 1000-bit 的序列, 长度≤ 500 的比特流的数量为: 1 + 2 + 4 + … + 25002^5002500 = 2501−12^501 - 12501−1, 远小于 210002^100021000, 即无法将 210002^100021000 个不同的元素映射到 2501−12^501 - 12501−1 个元素上.

Kolmogorov 复杂度

可理解的压缩算法: 将目标比特流 B 作为输入, 输出有用的、可读的 Java 代码 (该 Java 程序能够生成目标比特流 B).

最佳压缩算法: 将目标比特流 B 作为输入, 输出尽可能短的 Java 程序 (该 Java 程序能够生成目标比特流 B).

Java-Kolmogorov 复杂度 KJ(B)K_J(B)KJ​(B): 能够生成目标比特流的最短 Java 程序.

例如:

KJ(HugPlant.bmp)K_J(HugPlant.bmp)KJ​(HugPlant.bmp) = length(OptimalHP.java)length(OptimalHP.java)length(OptimalHP.java)

性质

性质一: 独立性.

Kolmogorov 复杂度完全独立于语言.

证明:

假设 KP(B)K_P(B)KP​(B) 为能够生成目标比特流的最短 Python 程序, 且 KP(B)K_P(B)KP​(B) 已知; KJ(B)K_J(B)KJ​(B) 为能够生成目标比特流的最短 Java 程序, KJ(B)K_J(B)KJ​(B) 未知.

可以用 Java 写 Python 解释器, 则 KJ(B)≤KP(B)+size(PythonInterpreter)K_J(B) \leq K_P(B) + size(Python Interpreter)KJ​(B)≤KP​(B)+size(PythonInterpreter).

由于用 Java 写的 Python 解释器的大小不会随目标比特流 B 变化, 所以可以认为是常数. 该做法同样适用于其他语言, 所以可以认为 Kolmogorov 复杂度完全独立于语言.

性质二: 不可计算性.

要编写一个程序来计算任何比特流的 Kolmogorov 复杂度是不可能的, 证明见此.

推论: 既然我们不能计算最短程序的长度, 也就不可能写出最优压缩算法.
