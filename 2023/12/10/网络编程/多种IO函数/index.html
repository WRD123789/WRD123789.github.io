MSG_OOB: 发送紧急消息

MSG_OOB 可选项用于处理带外数据. 带外数据是一种特殊的数据, 它在正常数据流之外传输, 用于发送紧急或优先级较高的信息.

收到 MSG_OOB 紧急消息时, 操作系统将产生 SIGURG 信号, 并调用注册的信号处理函数.

另外, 上述示例代码中的 fcntl(recv_sock, F_SETOWN, getpid()) 的含义是: “文件描述符 recv_sock 指向的套接字发出的 SIGURG 信号的处理进程变为 ID 为 getpid() 的进程”. 因为多个进程可以共同拥有 1 个套接字的文件描述符, 所以如果发生 SIGURG 信号, 无法确定让哪个进程来调用信号处理函数. 因此处理 SIGURG 信号时必须指定处理信号的进程.

紧急模式工作原理

TCP 紧急模式是一种机制, 用于在数据流中标识和处理紧急数据. 这个机制通过 TCP 头部的紧急指针字段来实现.

用于传输数据的 TCP 数据包的结构如下所示 (此处只关注紧急模式相关的信息, 其他不显示):

TCP 数据包中通常会包含:

  • 紧急指针: 它用于指示紧急数据的结束位置. 紧急指针字段包含一个16位的整数值, 表示相对于序列号的偏移量.
    • 例如: send(sock, "890", strlen("890"), MSG_OOB);, 则上述数据在输出缓冲区的分布如下:

      紧急指针指向偏移量为 3 的位置, 即紧急消息的下一个位置.
  • 紧急数据标记位: 只有该位被设置时, 紧急指针字段中包含的信息才有效.

当接收方在 TCP 连接上收到带有 URG 标志的数据时, 它会检查紧急指针字段, 找到紧急数据的结束位置.

检查输入缓冲

同时设置 MSG_PEEK 选项和 MSG_DONTWAIT 选项可以用来验证输入缓冲中是否存在接收数据. 设置 MSG_PEEK 选项并调用 recv 函数时, 即使读取了输入缓冲的数据也不会删除其中的数据, 因此该选项通常与 MSG_DONTWAIT 合作, 以非阻塞的方式验证输入缓冲中是否存在接收数据.

基于 Windows 的实现

由于在 Windows 中并不存在 Linux 那样的信号处理机制, 所以想要处理带外数据 (Out-of-band 数据) 只能使用 select 函数.

之前提到过, select 函数的 3 中监视项如下:

  • 是否存在套接字接收数据?
  • 无需阻塞传输数据的套接字有哪些?
  • 哪些套接字发生了异常?

其中 “异常” 是指不同寻常的程序执行流, 因此, 收到带外数据也属于异常, 因此在 Windows 平台上可以利用 select 函数的这一特性来处理带外数据.
