优于 select 的 epoll

利用 select 函数, 无论如何优化程序性能也无法同时接入上百个客户端, 因此 select 不适合以 Web 服务器端开发为主流的现代开发环境.

基于 select 的 I/O 复用技术速度慢的原因

观察基于 select 的 I/O 复用服务器端, 可以发现调用 select 函数后, 并不是将所有发生变化的文件描述符单独集中到一起, 而是通过观察作为监视对象的 fd_set 变量的变化, 找到发生变化的文件描述符, 因此此时无法避免针对所有监视对象的循环语句. 而且, 作为监视对象的 fd_set 变量在每次调用 select 函数之后都会发生变化, 所以每次调用 select 函数前, 都需要创造一份监视对象信息的副本, 然后通过 select 函数向操作系统传递监视对象信息的副本 (因为 select 需要操作系统的内核来检测是否有网络事件发生), 这些都是提高性能的障碍.

上述障碍中对性能影响最大的是「每次向操作系统传递监视对象信息」, 因为向操作系统传递监视对象信息需要从用户态切换到内核态, 会发生上下文切换, 而上下文切换的开销相比 for 循环通常较大.

想要解决上述问题, 可以通过如下方式弥补:

“仅向操作系统传递 1 次监视对象, 监视范围或内容发生变化时只通知发生变化的事项.”

这样就不需要每次调用 select 函数时都向操作系统传递监视对象信息, 但前提是操作系统支持这种处理方式. Linux 的支持方式是 epoll, Windows 的支持方式是 IOCP.

select 的优点

下面要讲的 epoll 只在 Linux 下提供支持, 也就是说, 改进的 I/O 复用模型不具有兼容性. 相反, 大部分操作系统都支持 select 函数. 因此, 只要满足如下条件, 即使在 Linux 平台也不应该拘泥于 epoll:

  • 服务器端接入者少.
  • 程序应具有兼容性.

实际不存在适用于所有情况的模型.

实现 epoll 时必要的函数和结构体

能够克服 select 函数缺点的 epoll 函数具有如下优点:

  • 无需编写以监视状态变化为目的的针对所有文件描述符的循环语句.
  • 调用对应于 select 函数的 epoll_wait 函数时无需每次传递监视对象信息.

下面介绍 epoll 服务器端实现中需要的 3 个函数:

  • epoll_create: 创建一个 epoll 实例, 该实例用来保存所有待监视的文件描述符. select 方式中为了保存监视对象文件描述符, 直接声明了 fd_set 变量. 但 epoll 方式下由操作系统负责保存监视对象文件描述符, 因此需要通过 epoll_create 函数向操作系统请求创建保存文件描述符的空间.
  • epoll_ctl: 控制 epoll 实例中的文件描述符. 为了添加和删除监视对象文件描述符, select 方式中需要使用宏 FD_SET、FD_CLR. 但在 epoll 方式中, 可以通过epoll_ctl 函数请求操作系统完成.
  • epoll_wait: 等待 epoll 实例中的文件描述符上的事件就绪.

select 方式中通过 fd_set 变量查看监视对象的状态变化, 而 epoll 方式中通过结构体 epoll_event 将发生变化的文件描述符集中到一起:

当声明足够大的 epoll_event 结构体数组后, 需要将它传递给 epoll_wait 函数, 发生变化的文件描述符信息会被填入该数组中.

水平触发和边缘触发

epoll 支持两种事件触发模式, 分别是边缘触发 (edge-triggered, ET) 和水平触发 (也称条件触发, level-triggered, LT).

对于水平触发, 只要被监控的 Socket 的输入缓冲中有数据, 就会一直通知该事件, 即每次调用 epoll_wait 返回的 “发生事件的文件描述符集合” 中都会包含该套接字对应的文件描述符, 直到 Socket 的输入缓冲中的数据读完为止. 因此, 水平触发只要满足事件的条件 (如套接字的输入缓冲中有数据可读), 就会不断将这个事件传递给用户进程.

对于边缘触发, 当被监控的 Socket 的输入缓冲中有数据时, 会通知一次该事件, 即使进程没有读取其中的数据, 也不会再次通知. 因此, 边缘触发只有在第一次满足条件的时候才会触发, 之后就不会再传递同样的事件.

实现边缘触发的回声服务器端

在实现边缘触发的回声服务器端之前, 首先需要了解下面两点:

  • 通过 errno 变量验证错误原因
  • 为了完成非阻塞 I/O, 更改套接字特性.

Linux 的套接字相关函数一般通过返回 -1 通知发生错误. 虽然知道发生了错误, 但仅凭这些内容无法得知产生错误的原因. 因此, 为了在发生错误时提供额外的信息, Linux声明了全局变量 int errno;.

为了访问该变量, 需要引入 error.h 头文件, 因为此头文件中有上述变量的 extern 声明. 另外, 每种函数发生错误时, 保存到 errno 变量中的值都不同, 没必要记住所有可能的值. 学习每种函数的过程中逐一掌握, 并能在必要时参考即可.

本节只介绍如下类型的错误: “read 函数发现输入缓冲中没有数据可读时返回 -1, 同时在 errno 中保存 EAGAIN 常量”.

如果使用边缘触发模式, I/O 事件发生时只会通知一次, 而且我们不知道到底能读写多少数据, 所以在收到通知后应尽可能地读写数据, 以免错失读写的机会. 因此, 我们会不断循环从文件描述符中读写数据, 那么如果文件描述符是阻塞的, 没有数据可读写时, 进程会阻塞在读写函数处, 程序就没办法继续往下执行. 所以, 边缘触发模式一般和非阻塞 I/O 搭配使用, 程序会一直执行 I/O 操作, 直到系统调用 (如 read 和 write) 返回错误 (当 read 函数返回 -1、变量 errno 中的值为 EAGIN 时, 说明套接字的输入缓冲中没有数据可读).

水平触发和边缘触发的优劣

边缘触发可以分离接收数据和处理数据的时间点, 避免了重复通知, 适用于高性能场景. 如果服务器接收到数据后暂时不想读取数据 (延迟处理), 那么使用边缘触发更合适, 因为对于水平触发, 如果服务器不读取套接字的输入缓冲中的数据, 那么每次调用 epoll_wait 函数时都会通知相应的事件, 随着事件数量的累加, 会给服务器带来一定的负担.

水平触发模型相对简单, 相比边缘触发更容易理解和使用, 也更稳定 (即使没有处理完所有事件, 也可以等到下一次 epoll_wait 返回时处理).
